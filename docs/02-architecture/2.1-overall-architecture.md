# 2.1 Overall Software Architecture

## Decision

Home Configurator is delivered as a suite:

```text
Frontend plugin
Optional Home Assistant integration
Model registry and asset packs
Portable presets
```

The canonical runtime is a full-screen Home Assistant custom view. A lightweight custom card is used only for preview, launching and simple dashboard integration.

## Runtime layers

1. Host adapter
2. Configuration kernel
3. Home Assistant translation layer
4. Application state
5. Experience shell
6. Capability controls
7. Persistent Three.js runtime
8. Model registry
9. Editor/setup application

## Data flow

```text
User input
→ interaction engine
→ semantic command
→ service adapter
→ Home Assistant
→ confirmed state
→ normalized state
→ presentation store
→ UI and renderer
```

## Deployment modes

- **Frontend-only:** dashboard config, standard HA states/services, built-in or URL models.
- **Full suite:** frontend plus integration-managed storage, discovery and model metadata.
- **Demo:** mocked HA adapter for documentation and testing.

## Architecture invariants

- Raw HA entities never reach UI or renderer packages.
- The integration enriches the frontend but is not required.
- Configuration and runtime state remain separate.
- 3D assets are data resolved through a registry.
- One failure must not crash the entire dashboard.
- Performance degrades by reducing effects before reducing interaction quality.
