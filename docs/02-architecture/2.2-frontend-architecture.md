# 2.2 Frontend Architecture

## Technology

- TypeScript
- Lit custom elements
- Three.js
- Vite/Rollup
- Framework-independent stores

## Runtime surfaces

- `home-configurator-view` — canonical full-screen application
- `home-configurator-card` — launcher and embedded preview
- `home-configurator-editor` — complete configuration application
- `home-configurator-demo` — mocked static host

## State ownership

- Config store
- Route store
- Overlay store
- Device state store
- Command store
- Presentation store
- Viewport store
- Accessibility store
- Diagnostics store

High-frequency gesture and animation values remain inside renderer/interaction controllers to avoid full component rerenders.

## Renderer lifecycle

One canvas is created per mounted view and retained across room and device transitions. Routes request scene changes; they never create their own renderer.

## Accessibility

Every gesture has a standard control alternative. Keyboard navigation, focus restoration, live announcements and reduced-motion behavior are first-class requirements.

## Build strategy

The initial shell is small. Renderer, editor, diagnostics and domain-specific controls are lazy-loaded where practical.
