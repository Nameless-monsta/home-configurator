# 2.4 Data Model

## Canonical document

```ts
interface HomeConfiguratorDocument {
  schemaVersion: number;
  id: string;
  name: string;
  metadata: DocumentMetadata;
  theme: ThemeConfig;
  navigation: NavigationConfig;
  performance: PerformanceConfig;
  floors?: FloorConfig[];
  rooms: RoomConfig[];
  systems?: SystemConfig[];
  modelSources?: ModelSourceConfig[];
}
```

## Main concepts

- **Floor:** optional organizational level.
- **Room:** usually maps to a Home Assistant area but can be manually composed.
- **Visual device:** one interaction unit that may bind one or many entities.
- **Entity binding:** semantic role such as `primary`, `power`, `battery` or `source`.
- **Capability:** normalized function such as brightness, target temperature or media playback.
- **Control:** interaction component bound to a capability.
- **Presentation:** model, camera, lighting, motion, material and exploded-view metadata.

## Runtime separation

Persisted configuration does not contain active route, gesture, animation or optimistic-command state.

Runtime models include:

- resolved installation config
- normalized device state
- optimistic patches
- pending operations
- serializable route state
- presentation state

## Rules

- Every configurable object has a stable ID.
- Ordering is explicit and deterministic.
- Entity references are preserved when missing.
- JSON is canonical storage; YAML is supported for user import/export.
- No credentials, executable scripts or embedded binary models are permitted.
- Schemas, presets, models and APIs have independent version tracks.
