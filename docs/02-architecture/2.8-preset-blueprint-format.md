# 2.8 Preset / Blueprint Format

## Goal

Make complete Home Configurator experiences portable without tying them to one installation's entity IDs.

## Principle

Blueprints export intent and mapping requirements rather than fixed bindings.

```yaml
placeholder: living_room_main_light
expected_profiles:
  - light_rgb
required_capabilities:
  - power
  - brightness
optional_capabilities:
  - color
```

## Import flow

```text
Import
→ validate schema and dependencies
→ run discovery
→ auto-match placeholders
→ classify high/medium/low confidence
→ review conflicts and unmatched requirements
→ generate installation document
```

## Blueprint content

- metadata and compatibility
- theme/design tokens
- navigation and layout intent
- room and system templates
- visual-device templates
- capability requirements
- model recommendations
- camera, lighting and motion presets
- quick actions
- namespaced extensions

## Export levels

- theme only
- presentation only
- layout/navigation
- interaction mappings
- complete portable experience

## Security

Blueprints cannot contain credentials, arbitrary scripts, untrusted inline shader code or embedded binary assets. Dependencies are declared and validated before application.
