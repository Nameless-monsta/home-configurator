# 2.9 HACS Packaging Strategy

## Decision

HACS is the installer, not the runtime architecture.

## Initial repositories

V1 may begin with two HACS-installable repositories:

1. **Frontend dashboard plugin** — custom view, card, editor and core runtime.
2. **Optional integration** — persistent storage, optimized discovery and model metadata.

A small generic model library is bundled with the frontend. Large or manufacturer-specific packs can be split later.

## Package families

Future package categories may include:

- model packs
- theme packs
- preset packs
- environment packs
- capability/profile plugins

Shader and animation extensions must remain behind validated registries and compatibility contracts.

## Requirements

Every package declares:

- ID, type and version
- author and license
- minimum/maximum core version
- dependencies and optional dependencies
- assets and integrity metadata
- required capabilities

## Operational rules

- Core operation is local and offline after installation.
- No mandatory CDN, analytics or cloud service.
- Packages version independently.
- Dependency graphs remain acyclic.
- Invalid or incompatible packages refuse to load safely.
- The runtime continues with fallbacks when optional content is missing.
