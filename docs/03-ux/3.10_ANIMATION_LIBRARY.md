# 3.10 — Animation Library

**Status:** Approved baseline

## Purpose

The animation library defines reusable motion behaviors for objects, materials, controls, feedback and ambient state. Transitions move the application between states; animations describe behavior within a state.

All animations must reference HCML and HCDS tokens. Feature modules may select and parameterize registered animations but may not introduce independent motion systems.

## Core principles

1. Motion communicates state, capability or causality.
2. The 3D device remains the main animated subject.
3. UI motion is restrained and subordinate.
4. Ambient animation pauses or reduces when it no longer adds meaning.
5. User-driven motion responds immediately and settles physically.
6. Confirmed Home Assistant state determines final animation state.
7. Device capability controls which animations are available.
8. Motion remains physically plausible for the represented object.
9. Reduced-motion mode removes nonessential movement.
10. Every continuous animation has a lifecycle and performance budget.

## Animation registry contract

```ts
interface AnimationDefinition {
  id: string;
  family: AnimationFamily;
  targets: AnimationTargetType[];
  trigger: AnimationTrigger;
  duration?: MotionToken;
  loop?: boolean;
  interruptible: boolean;
  reducedMotionFallback: string;
  qualityMinimum?: "full" | "balanced" | "essential";
}
```

Animation triggers:

```text
state entry
state exit
capability value
user gesture
command pending
command confirmed
command failed
ambient idle
system event
```

## Motion families

### AN-01 Hero idle

Purpose: keep the selected object alive without making it float like a screensaver.

Behavior:

- near-imperceptible breathing or orientation drift;
- amplitude depends on object type and material;
- pauses during direct manipulation;
- stops in reduced-motion mode;
- never changes the meaning of position or control value.

### AN-02 Object manipulation

Used for drag rotation and approved pinch/zoom behavior.

Stages:

```text
capture
→ direct response
→ constrained movement
→ release
→ inertia
→ settle
```

Rules:

- no input smoothing that creates visible lag;
- constraints apply progressively, not as abrupt walls;
- inertia depends on release velocity;
- settle returns only to the nearest valid resting state.

### AN-03 Magnetic settle

Used when an object, selector or control value approaches a meaningful stop.

Examples:

- camera hero orientation;
- mode selector;
- scene option;
- exploded-view endpoint.

It uses a restrained spring with no cartoon bounce.

### AN-04 Power state

Power on:

- object material wakes;
- emission rises from the physical source;
- local reflected light follows;
- related UI state confirms.

Power off reverses the sequence with a shorter emission tail where physically plausible.

### AN-05 Brightness response

Brightness controls:

- emissive intensity;
- local environment contribution;
- small readout value;
- optional bloom within quality budget.

Brightness animation is continuous and linear during gesture preview, then gently reconciles to confirmed state.

### AN-06 Color response

- device emission hue changes continuously;
- reflected light follows with restrained interpolation;
- neutral interface elements remain unchanged;
- saturation at very low brightness remains visually legible without falsifying the selected value.

### AN-07 Color sphere behavior

Registered sub-animations:

- sphere rotation under one-finger drag;
- selected point tracking;
- luminosity halo response;
- two-finger brightness expansion/contraction cue;
- release settle;
- unavailable desaturation.

The sphere must feel volumetric, not like a flat color wheel wrapped in a circle.

### AN-08 Climate airflow

Modes:

```text
cooling
heating
dry
fan only
auto
idle
```

Behavior:

- airflow originates from a plausible vent or thermostat visualization anchor;
- direction reflects swing/fan state where available;
- speed reflects fan mode;
- color remains restrained and semantic;
- idle retains only minimal environmental indication.

Tier C uses a simple directional field or material cue rather than particles.

### AN-09 Temperature adjustment

- dial/ring follows pointer directly;
- target temperature readout interpolates continuously;
- ambient thermal field previews the target;
- release dispatches or confirms according to command policy;
- confirmed value settles without a snap.

### AN-10 Media playback

Playing:

- screen/content illumination active;
- subtle playback progress movement;
- optional audio-reactive behavior only when actual data is available.

Paused:

- content remains visible;
- reactive motion stops;
- active state remains clear.

Buffering:

- localized progress indicator;
- no full-device pulsing.

### AN-11 Volume response

- readout follows continuous gesture;
- speaker or system visualization may expand in a controlled radial or vertical response;
- mute collapses emission/energy cue without hiding current volume memory;
- confirmed value reconciles smoothly.

### AN-12 Vacuum idle

- lidar or small status element may rotate only if representative of the real device/state;
- docked state is quiet;
- battery/status halo remains subtle;
- no roaming animation while the physical device is idle.

### AN-13 Vacuum cleaning

- wheel/brush animation activates;
- route or cleaning trace may animate when map data exists;
- selected room/zone receives controlled emphasis;
- battery and progress remain secondary;
- pause freezes movement without resetting route context.

### AN-14 Vacuum return/dock

- route intent changes toward dock;
- hero orientation may align with dock representation;
- docked confirmation ends motion cleanly;
- failure uses localized feedback rather than shaking the whole device.

### AN-15 Scene preview

- affected device/environment values preview together;
- preview remains clearly provisional;
- cancel returns to confirmed state through the shortest coherent path;
- activate transitions into confirmed-state reconciliation.

### AN-16 Scene activation wave

Used to communicate coordinated change across several devices.

- wave order follows spatial or configured sequence;
- duration is capped;
- device-specific changes remain accurate;
- the effect does not delay service calls.

### AN-17 Exploded view

```text
prepare
→ parts separate
→ labels appear
→ inspection idle
→ labels leave
→ parts reassemble
```

Rules:

- authored part offsets only;
- part order follows functional hierarchy;
- camera reframes if needed;
- labels remain UI/HTML unless true spatial anchoring is required;
- reassembly is reversible.

### AN-18 Model assemble/reveal

Used for first meaningful reveal or selected product transitions.

- parts may resolve from a compressed origin or authored assembly path;
- use sparingly;
- never replay on every routine visit;
- Tier C replaces with matched fade/scale reveal.

### AN-19 Material morph

Properties supported:

```text
base color
roughness
metalness
transmission
emissive color
emissive intensity
opacity
```

Morphing must avoid impossible intermediate values and respect material class.

### AN-20 Pending command

- targeted control or device part shows a small pending state;
- optimistic value remains visible;
- repeated commands coalesce;
- no global spinner for local actions.

### AN-21 Confirmation

- control settles into confirmed value;
- pending marker clears;
- optional micro-highlight appears once;
- success is not celebrated with unnecessary motion.

### AN-22 Command failure

- optimistic state returns to confirmed state;
- affected control receives restrained directional correction or border/text feedback;
- error message explains the failure;
- no aggressive shake unless the control metaphor explicitly requires it.

### AN-23 Availability loss/recovery

Loss:

- active motion ceases;
- emission reduces;
- material adopts unavailable modifier;
- control affordances dim and disable.

Recovery:

- confirmed state restores;
- capability-specific motion resumes only if appropriate.

### AN-24 Navigation row motion

- rows enter with very short restrained stagger;
- active row marker moves continuously;
- hover/focus uses opacity, line and slight translation;
- no bouncing icons or oversized scaling.

### AN-25 Control rail motion

- rail reveals from a stable viewport anchor;
- controls enter in capability priority order;
- changing controls morphs or replaces only the affected region;
- rail does not repeatedly slide in during small state updates.

### AN-26 Status indicator

States:

```text
connected
pending
disconnected
error
reconnecting
```

Motion remains tiny and stops when status is stable.

### AN-27 Loading

Loading hierarchy:

1. preserve last valid frame;
2. show proxy or silhouette;
3. use localized progress;
4. reveal ready content.

Avoid indefinite rotating spinners in the center of the stage.

### AN-28 Ambient environment

Permitted:

- subtle grain movement;
- device-generated glow drift;
- sparse semantic particles;
- slow procedural field.

Prohibited:

- permanent decorative network lines;
- dense particles unrelated to state;
- outer-space motion;
- continuous high-amplitude background color changes.

### AN-29 Focus and selection

- focus-visible ring appears immediately;
- selected marker travels between items when possible;
- selection animation does not obscure the label;
- keyboard and touch selections use the same final state.

### AN-30 Notification and discovery cue

Used for new devices, missing mappings and editor review.

- compact badge or marker enters once;
- no repeated pulse after acknowledgment;
- runtime navigation is not rearranged through animation.

## Duration classes

```text
micro feedback: 100–180 ms
control response: 160–280 ms
settle: 260–480 ms
object state: 350–700 ms
ambient cycle: 4–16 s
continuous: capability-driven
```

## Spring classes

### S1 — precise

Controls, markers and small UI elements.

### S2 — weighted

Hero object settle and camera-adjacent movement.

### S3 — soft environmental

Lighting and atmosphere reconciliation.

No spring class may produce repeated visible oscillation in normal operation.

## Loop policy

Continuous loops require:

- semantic purpose;
- visibility detection;
- pause when view is hidden;
- reduced-motion fallback;
- quality-tier minimum;
- bounded resource use.

Loops are stopped when the relevant device is no longer active.

## Synchronization

Animation clocks:

```text
UI clock
renderer clock
media/progress clock
confirmed state timestamps
```

The transition director may synchronize families, but media progress must derive from source state rather than visual animation time alone.

## Responsive behavior

- phone layouts reduce simultaneous animated elements;
- iPad landscape receives the full intended composition;
- desktop hover motion remains optional and restrained;
- orientation changes reflow rather than replay entry animation.

## Reduced motion

- hero idle stops;
- particles and ambient flow stop or become static;
- assembly/explode uses compact direct transforms;
- stagger is removed;
- confirmation uses opacity/line changes;
- value controls remain directly responsive.

## Performance tiers

### Full

- complete material/lighting response;
- approved particles and post-processing;
- authored part animation;
- smooth environment interpolation.

### Balanced

- fewer particles;
- simplified shadows and bloom;
- reduced material layers;
- lower ambient update rate.

### Essential

- direct transforms and opacity;
- simple emissive/material state;
- no expensive procedural effects;
- object and control responsiveness preserved.

## Accessibility

- motion never carries the sole meaning;
- live values expose accessible text/ARIA values;
- looping motion pauses under reduced-motion preferences;
- focus remains stable during animation;
- disabled/unavailable state is announced;
- animation completion is not required before the next keyboard action unless state integrity demands it.

## Developer diagnostics

Developer mode can display:

- active animation IDs;
- target nodes/materials;
- trigger and capability source;
- elapsed duration and progress;
- loop status;
- spring velocity;
- quality tier;
- reduced-motion replacement;
- dropped frames and animation budget.

## Acceptance criteria

1. All runtime animation uses registered families.
2. Hero idle is subtle, pausable and optional.
3. Direct manipulation has immediate response, inertia and constrained settle.
4. Power, brightness and color animate the physical source and environment coherently.
5. Climate, media, vacuum and scene behaviors have domain-specific motion.
6. Pending, confirmed, failed and unavailable command states are defined.
7. Loading preserves a meaningful frame or proxy.
8. Exploded-view and model assembly are authored and reversible.
9. Continuous loops have lifecycle and performance controls.
10. Reduced-motion variants exist for all nonessential movement.
11. Full, balanced and essential tiers preserve the same state meaning.
12. Motion never becomes the only communication channel.
13. Focus and keyboard behavior remain stable during animation.
14. Diagnostics can identify active animations and performance cost.
