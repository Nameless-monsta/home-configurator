# 3.1 Navigation Shell

**Status:** Complete baseline specification  
**Phase:** 3 — UX & Visual Design  
**Depends on:** Stage 1 interaction specification, HCDS, HCML, Phase 2 architecture

## 3.1.1 Objective

Define the persistent navigation shell that frames the entire Home Configurator experience without behaving like a conventional dashboard header.

The shell must:

- remain visually restrained;
- preserve the central 3D stage;
- provide immediate access to rooms, devices and systems;
- support iPad landscape as the canonical layout;
- adapt cleanly to iPhone portrait and landscape;
- open menus as cinematic interface states rather than ordinary dropdowns;
- preserve current scene context during navigation;
- expose connection, availability and pending-state feedback without clutter;
- remain usable by touch, keyboard and assistive technology.

---

## 3.1.2 Shell composition

```text
Navigation Shell
├── Brand / root control
├── Context path
├── Adaptive navigator trigger
├── Current state summary
├── Previous / next control
├── Connection / pending indicator
└── Overlay host
```

The shell is persistent across:

- home;
- room;
- device;
- control;
- system;
- unavailable and loading states.

The WebGL stage remains mounted beneath it.

---

## 3.1.3 Canonical desktop and iPad layout

```text
┌─────────────────────────────────────────────────────────────┐
│ Brand      Room / Device Context          Navigator   Status │
│                                                             │
│                                                             │
│                       3D STAGE                              │
│                                                             │
│                                                             │
│ Previous                                         Next       │
└─────────────────────────────────────────────────────────────┘
```

### Left zone

Contains:

- root/brand control;
- current room and device identity;
- optional short state summary.

### Centre zone

Reserved for:

- adaptive navigator trigger;
- current category title in expanded desktop mode.

### Right zone

Contains:

- pending or disconnected state;
- optional quick action;
- menu state indicator.

### Bottom navigation zone

Contains:

- previous/next device controls;
- current index where useful;
- no permanent bottom tab bar.

## 3.1.4 Primary shell states

```text
SHELL_IDLE
SHELL_FOCUSED
SHELL_NAV_OPEN
SHELL_TRANSITIONING
SHELL_PENDING
SHELL_DISCONNECTED
SHELL_ERROR
SHELL_COMPACT
SHELL_HIDDEN_UI
```

### SHELL_IDLE

- minimal typography;
- no raised surfaces unless required for contrast;
- current room/device visible;
- navigator trigger compact;
- scene fully interactive.

### SHELL_FOCUSED

- active control or device receives emphasis;
- inactive shell text reduces in opacity;
- contextual status becomes visible;
- stage interaction remains enabled unless a control captures input.

### SHELL_NAV_OPEN

- scene remains mounted;
- scene interaction is suspended;
- navigation overlay moves to Z4;
- shell trigger morphs into close/back state;
- focus moves into the overlay;
- current room/device remains legible.

### SHELL_TRANSITIONING

- pointer input is selectively locked;
- route and model transitions are coordinated;
- shell labels transition after object departure begins;
- status remains available.

### SHELL_PENDING

- small pending-state marker appears;
- optimistic visual state continues;
- navigation remains available unless command safety requires locking.

### SHELL_DISCONNECTED

- connection status becomes explicit;
- scene remains visible using last confirmed state;
- device controls are disabled or queued according to policy;
- navigation remains operational.

### SHELL_ERROR

- error is contextual rather than full-screen where possible;
- shell offers recovery or back action;
- unaffected rooms/devices remain accessible.

### SHELL_COMPACT

- used on iPhone and constrained embeds;
- context path shortens;
- navigator becomes a single pill;
- previous/next controls move to bottom dock.

### SHELL_HIDDEN_UI

- optional temporary inspection state;
- shell fades to minimal markers;
- any tap outside the object restores the shell;
- must never hide critical connection or error information.

## 3.1.5 Adaptive navigator trigger

The main trigger is a compact, text-led control.

It contains, depending on profile:

```text
Current room
Current device
Menu index
Chevron / menu morph
Optional count
```

### Behavior

Tap:

```text
Idle
→ trigger expands slightly
→ background darkens locally
→ navigation overlay unfolds
→ rows reveal with stagger
```

Tap while open:

```text
Open
→ rows collapse
→ overlay recedes
→ trigger returns to idle
```

Long press is not required for core navigation.

---

## 3.1.6 Navigation overlay hierarchy

The overlay contains three primary modes:

```text
ROOMS
DEVICES
SYSTEMS
```

Optional secondary modes:

```text
SCENES
FAVORITES
SETUP
```

The default open mode depends on context:

- from home: rooms;
- from room: devices;
- from device: devices with room context retained;
- from system: systems.

The user can switch modes without closing the overlay.

```text
┌─────────────────────────────────────────────────────────────┐
│ Context / close                                             │
│                                                             │
│ ROOMS     DEVICES     SYSTEMS                               │
│                                                             │
│ 01 Living Room                                              │

│ 04 Hallway                                                  │
│                                                             │
│ Supporting metadata / current state                         │
└─────────────────────────────────────────────────────────────┘
```

### Primary list

- large text;
- generous row height;
- index aligned consistently;
- selected item visibly marked;
- optional count or short state summary.

### Secondary metadata

May show:

- active devices;
- availability;
- current temperature;
- playing media;
- cleaning status;
- lights on count.

It must not become a dense dashboard summary.

## 3.1.8 Room row behavior

A room row includes:

```text
Index
Room name
Optional short state
Current marker
Optional device count
```

Tap room:

```text
Row acknowledges
→ current scene de-emphasizes
→ overlay begins departure
→ camera/object transition begins
→ room route commits
→ room default device or room overview enters
```

The overlay does not disappear before the transition has visibly started.

---

## 3.1.9 Device row behavior

A device row includes:

```text
Index
Device name
Profile label or short state
Availability marker
Current marker
```

Examples of short state:

```text
ON · 62%
PLAYING · Apple TV
COOLING · 22 °C
CLEANING · 48%
CLOSED
UNAVAILABLE
```

No device row exposes all entity attributes.

---

## 3.1.10 System navigation

Generated systems may include:

- lighting;
- climate;
- media;
- cleaning;
- security;
- energy.

System rows lead to cross-room experiences rather than lists of cards.

Example:

```text
Lighting
→ spatial overview of active lights
→ select device
→ transition into device configurator
```

---

## 3.1.11 Quick access mode

Quick access is a smaller overlay or dropdown for frequent navigation.

Contents are ranked by:

1. explicit favorites;
2. current room devices;
3. recent interaction;
4. room defaults;
5. active or important devices.

Maximum visible items:

- iPad/desktop: 6–8;
- iPhone landscape: 5–6;
- iPhone portrait: 4–5.

The quick-access view must not duplicate the full navigation overlay.

---

## 3.1.12 Previous and next controls

Previous/next navigates within the current context.

Examples:

- within a room: previous/next device;
- within systems: previous/next system item;
- within exploded view: previous/next component where enabled.

Behavior:

- controls use text or minimal arrow marks;
- the next item may be previewed subtly;
- hold is not required;
- disabled state is visible at collection boundaries unless wrapping is enabled.

Default wrapping:

```text
Room device list: enabled
System list: enabled
Setup/editor sequences: disabled
```

---

## 3.1.13 Context path

Examples:

```text
Living Room / Floor Lamp
Bedroom / Climate
Cleaning / Roborock
```

Rules:

- room and device are separated by a slash;
- no browser-style breadcrumbs with multiple clickable chips;
- room portion may be clickable on desktop/iPad;
- iPhone portrait may show only the active device with room in secondary text;
- user-configured names always take precedence.

---

## 3.1.14 Status indicators

Allowed persistent status:

- Home Assistant disconnected;
- device unavailable;
- command pending;
- configuration invalid;
- discovery change pending review.

Status treatment:

- small text or dot;
- monochrome by default;
- restrained semantic color only for persistent warning/error;
- no notification badge clutter.

---

## 3.1.15 Motion specification

The shell follows HCML.

### Open navigation

```text
0–80 ms      trigger acknowledgment
60–260 ms    overlay veil/blur enters
120–420 ms   overlay structure unfolds
220–620 ms   rows reveal with 28–42 ms stagger
420–700 ms   metadata settles
```

### Close navigation

```text
0–180 ms     rows reduce opacity and shift
100–360 ms   overlay recedes
220–480 ms   trigger contracts
after 480 ms stage input restores
```

### Route selection

```text
0–120 ms     selected row acknowledgment
80–360 ms    existing object de-emphasis
160–520 ms   overlay departure
260–980 ms   camera/object transition
720–1180 ms  new shell context enters
```

No spring overshoot on the full-screen overlay.

Small trigger controls may use subtle spring settling.

---

## 3.1.16 Responsive behavior

### iPad landscape

Canonical mode:

- full shell;
- quick navigator available;
- large overlay text;
- previous/next at lower edges;
- no page scrolling.

### Desktop

- expanded context allowed;
- optional simultaneous room/device column in full overlay;
- hover may preview but never gate interaction;
- keyboard navigation required.

### iPhone landscape

- compact top pill;
- full-width dropdown/overlay;
- one list mode visible at a time;
- bottom previous/next dock;
- context title shortened.

### iPhone portrait

- minimal top pill;
- full-screen navigation overlay;
- bottom safe-area controls;
- one navigation level visible at a time;
- large rows;
- no multi-column layout.

## 3.1.17 Touch targets

Minimum touch target:

```text
44 × 44 px
```

Preferred navigation row height:

```text
iPad/Desktop: 56–72 px
iPhone: 52–64 px
```

The visual mark may be smaller than the hit area.

---

## 3.1.18 Keyboard behavior

Required keys:

```text
Tab / Shift+Tab  move focus
Enter / Space    activate
Escape           close overlay / step back
Arrow Up/Down    move through rows
Arrow Left/Right switch navigation mode or previous/next
Home / End       first/last item
```

Focus returns to the navigator trigger after overlay close.

---

## 3.1.19 Accessibility behavior

- overlay uses dialog/navigation semantics;
- active room/device uses `aria-current`;
- pending and disconnected states use live regions;
- row state is announced concisely;
- focus is trapped only in full overlay states;
- reduced-motion mode removes stagger and shortens transitions;
- menu close remains available as a standard button.

---

## 3.1.20 Empty and error states

### No rooms

```text
No configured rooms
Review discovered areas
```

### Room with no accepted devices

```text
No devices shown
Review discovered devices
```

### Missing device

- row remains in place if configured;
- marked unavailable;
- opens a recovery/detail state rather than disappearing.

### Invalid route

- returns to nearest valid room or home;
- preserves an error diagnostic;
- never leaves a blank stage.

---

## 3.1.21 Visual rules

- background remains `#101010`;
- no permanent green or colored accents;
- hairlines are rare and subtle;
- text is the dominant navigation affordance;
- icons are secondary;
- overlay surfaces are near-opaque on small screens;
- no card grid;
- no floating orb or decorative sphere in the background;
- no unrelated network lines;
- selected state uses typography, marker, motion or fill—not neon borders.

## 3.1.24 Acceptance criteria

3.1 is approved when:

1. The shell remains mounted across all routes.
2. The central 3D object remains visually dominant.
3. Rooms, devices and systems are reachable within two interactions.
4. Full navigation opens without remounting the stage.
5. Overlay behavior is defined on iPad, iPhone and desktop.
6. Touch, keyboard and assistive technology paths are complete.
7. Pending and disconnected states remain visible but restrained.
8. Previous/next behavior is contextual and deterministic.
9. No navigation state falls back to Lovelace cards or a permanent tab bar.
10. The shell follows HCDS and HCML without inventing new visual rules.
11. Empty, missing and invalid-route states are recoverable.
12. The specification is sufficient to proceed to room navigation and high-fidelity prototyping.

---

## 3.1.25 Decision

The Home Configurator navigation shell is a persistent, text-led, cinematic control layer over one continuous 3D world.

It is not a header, sidebar, tab bar or collection of dashboard controls.
