# 3.2 Room Navigation

**Status:** Complete  
**Phase:** 3 — UX and Visual Design  
**Depends on:** 3.1 Navigation Shell, 1.5 Navigation State Machine, 1.6 Motion Language, 1.8 Responsive Behaviour, 1.9 Camera and Lighting

## 3.2.1 Objective

Define how users discover, enter, leave, switch, preview and manage rooms without breaking the feeling of one continuous spatial environment.

Room navigation must feel like moving through one connected home, not loading separate dashboard pages.

## 3.2.2 Core principle

```text
Room selection
→ spatial transition
→ room context resolves
→ default device or room overview appears
→ controls adapt
```

The room route changes, but the application shell and WebGL stage remain mounted.

## 3.2.3 Room states

```text
UNSELECTED
PREVIEWING
ENTERING
ACTIVE
SWITCHING_OUT
SWITCHING_IN
EMPTY
UNASSIGNED
UNAVAILABLE
```

### UNSELECTED

No room context is active. Used only on initial home overview or setup.

### PREVIEWING

The user is highlighting a room in navigation but has not committed selection.

### ENTERING

Camera, scene, title and device context are transitioning into the room.

### ACTIVE

The room is fully selected and its default device or overview is available.

### SWITCHING_OUT / SWITCHING_IN

Intermediate states during room-to-room movement.

### EMPTY

The room exists but has no accepted controllable devices.

### UNASSIGNED

The room represents devices or entities without a resolved area.

### UNAVAILABLE

The room configuration exists but all visible devices are unavailable.

## 3.2.4 Room entry points

Users may enter a room from:

- the full navigation overlay;
- the quick-access menu;
- previous/next room controls;
- a room object in the home overview;
- a deep link;
- a scene or automation context;
- restored session state;
- automatic first-run default route.

All entry points resolve to the same room transition pipeline.

## 3.2.5 Room selection model

```ts
interface RoomSelectionIntent {
  roomId: string;
  source:
    | "full-navigation"
    | "quick-access"
    | "previous-next"
    | "home-overview"
    | "deep-link"
    | "history"
    | "system"
    | "initial";
  preferredDeviceId?: string;
}
```

The route coordinator validates the target room before starting visual movement.

## 3.2.6 Room hierarchy

Preferred hierarchy:

```text
Installation
├── Floor
│   ├── Room
│   │   ├── Default device
│   │   ├── Other devices
│   │   └── Scenes
│   └── Room
└── Unassigned
```

Floors are optional. Small homes should never be forced to expose floor navigation.

## 3.2.7 Room route

Canonical route:

```text
/room/:roomId
```

Optional room-and-device route:

```text
/room/:roomId/device/:deviceId
```

The room route must remain serializable, deep-linkable and restorable.

## 3.2.8 Room overview composition

When a room is active and no device is explicitly selected, the room overview displays:

- room name;
- optional floor name;
- room status summary;
- room-level 3D object or abstract spatial composition;
- default device emphasis;
- available device count;
- active scene or environment state where useful;
- next/previous room navigation;
- quick device access.

The overview must not become a tile grid.

## 3.2.9 Default device resolution

Priority:

```text
1. Explicit room defaultDeviceId
2. Last selected device for the room
3. Highest-priority accepted controllable device
4. Room overview
```

Rules:

- unavailable devices are skipped unless explicitly requested;
- remembered devices are session-level by default;
- a room with multiple devices may remain in overview rather than forcing one device;
- user override always wins.

## 3.2.10 Room preview

Preview is available in expanded navigation and pointer-capable layouts.

Preview may show:

- room name;
- floor;
- device count;
- active device count;
- temperature or climate summary;
- lights-on count;
- a restrained thumbnail or live stage framing;
- availability warning.

Preview must not trigger Home Assistant services or permanently alter route state.

## 3.2.11 Room transition choreography

Baseline room switch:

```text
0–90 ms
Selection acknowledges input

70–280 ms
Current room typography and controls de-emphasize

120–520 ms
Current room object/scene begins authored departure

260–860 ms
Camera travels through transition path

480–980 ms
Target room environment and object enter

760–1120 ms
Room title, status and default device settle

After 1120 ms
Idle room motion resumes
```

Typical total duration:

```text
0.95–1.25 s
```

Longer transitions are allowed only for meaningful floor changes or authored spatial effects.

## 3.2.12 Spatial continuity rules

Room changes should preserve continuity through at least one shared cue:

- camera direction;
- object movement direction;
- light/environment continuity;
- floor/plan geometry;
- typography anchor;
- transition veil;
- room-to-room adjacency metadata.

Avoid arbitrary dissolve-only transitions for every room.

## 3.2.13 Adjacency model

Optional room adjacency metadata:

```ts
interface RoomAdjacency {
  fromRoomId: string;
  toRoomId: string;
  direction?: "left" | "right" | "forward" | "back" | "up" | "down";
  transitionPreset?: string;
}
```

When absent, the transition director uses list order and generic direction rules.

## 3.2.14 Same-floor switching

Preferred behavior:

- lateral or shallow-depth camera movement;
- minimal environment discontinuity;
- similar camera elevation;
- room title anchor remains stable;
- transition direction follows adjacency or room order.

## 3.2.15 Cross-floor switching

Preferred behavior:

- vertical travel cue;
- slightly longer camera path;
- floor label appears briefly;
- room controls remain suppressed until target floor resolves;
- transition must not resemble a page reload.

Cross-floor duration target:

```text
1.15–1.55 s
```

## 3.2.16 Room-to-device handoff

After a room becomes active:

```text
Room context resolves
→ default device selected or overview retained
→ device model prepares
→ room-level lighting persists
→ device controls appear
```

Room transition completes before detailed device manipulation becomes active.

## 3.2.17 Device list inside room

The room device list is presented as:

- a minimal vertical list in full navigation;
- a short contextual selector in the room view;
- previous/next device controls;
- optional spatial labels around the active composition.

It must not present standard HA entity cards.

## 3.2.18 Room status summary

Possible room summary fields:

```text
Lights: 3 on
Climate: 22.5 °C · Cooling
Media: Playing
Cleaning: Idle
Windows: Closed
Presence: Occupied
```

Rules:

- show only meaningful, high-confidence data;
- limit default room summary to 2–4 items;
- diagnostic entities remain hidden;
- values update live without re-entering the room.

## 3.2.19 Empty room

An empty room remains visually intentional.

Required behavior:

- retain room name and shell;
- display a quiet room placeholder, not an error;
- provide "Add devices" in editor-enabled contexts;
- preserve room navigation;
- do not auto-delete the room.

## 3.2.20 Unassigned room

The unassigned room is a special review context.

It may contain:

- devices without area assignment;
- entities with conflicting area data;
- devices awaiting admission;
- low-confidence synthetic groups.

Runtime visibility is configurable.

Editor behavior should prioritize review and assignment.

## 3.2.21 All-devices-unavailable room

When all visible devices are unavailable:

- room remains selectable;
- room presentation desaturates subtly;
- controls are disabled selectively;
- room summary states "Devices unavailable";
- navigation remains functional;
- no blocking error dialog appears automatically.

## 3.2.22 Newly discovered room

A newly discovered HA area may become:

```text
accepted automatically
review required
hidden
```

Recommended default:

- auto-accept when it contains at least one high-confidence controllable device;
- review when area or grouping confidence is medium/low;
- preserve order of existing rooms.

## 3.2.23 Room ordering

Order priority:

```text
1. Explicit user order
2. Floor order
3. Discovery order
4. Locale-aware name
5. Stable room ID
```

Room reordering never changes entity assignment.

## 3.2.24 Previous and next room behavior

The previous/next controls follow resolved visible room order.

Rules:

- hidden rooms are skipped;
- unavailable rooms remain included unless configured otherwise;
- wrapping is configurable;
- controls show target room names on hover/focus where space allows;
- repeated rapid input queues only the latest target.

## 3.2.25 Swipe navigation

Optional swipe navigation may be enabled on touch devices.

Gesture rules:

- horizontal edge swipe only;
- disabled while manipulating a 3D object or continuous control;
- minimum travel threshold;
- velocity threshold required;
- visual preview follows finger only after intent is clear;
- cancellation returns to the current room with spring settle.

Default recommendation:

```text
Off in v1 unless interaction testing proves it conflict-free
```

## 3.2.26 Room quick access

Quick access can include:

- favorite rooms;
- recently used rooms;
- rooms with active alerts;
- room defaults configured by the user.

Automatic ranking must not permanently reorder navigation.

## 3.2.27 Responsive behavior

### iPad landscape

- room name and context remain visible;
- room selector opens as dropdown/overlay from top shell;
- stage stays fully visible;
- room overview may show device shortcuts along one edge;
- no scrolling required for primary room selection.

### iPhone landscape

- room name remains compact;
- room list uses full-width overlay;
- only one secondary context group appears at a time;
- next/previous room controls remain accessible.

### iPhone portrait

- room selector opens full-screen;
- active room title uses compact top treatment;
- room summary collapses to one or two values;
- device access moves to bottom dock or contextual sheet.

### Desktop

- expanded navigation may preview adjacent rooms;
- keyboard navigation is fully supported;
- hover preview is allowed but not required.

## 3.2.28 Keyboard behavior

Suggested bindings:

```text
Arrow Left / Right: previous or next room when focus is in room navigator
Enter / Space: select highlighted room
Escape: close room navigator
Home / End: first or last room
Page Up / Page Down: previous or next floor where floors exist
```

Global shortcuts must not interfere with focused form controls.

## 3.2.29 Accessibility

Requirements:

- room selector exposes listbox or menu semantics;
- current room is announced;
- room counts and availability states are accessible;
- transition completion moves focus predictably;
- room preview is not required to understand selection;
- swipe has button alternatives;
- reduced-motion mode shortens spatial travel and removes parallax;
- floor changes are announced.

## 3.2.30 Reduced-motion behavior

When reduced motion is enabled:

- camera transitions use shorter direct interpolation;
- object departure/arrival uses opacity and small positional change;
- no long parallax path;
- floor changes use restrained fade/shift;
- focus and route updates remain identical.

Target duration:

```text
180–320 ms
```

## 3.2.31 Loading behavior

If target room assets are not ready:

```text
Room route commits
→ shell/title update
→ proxy room/device presentation appears
→ model loads
→ authored replacement transition
```

Never block room navigation behind a full-screen spinner.

## 3.2.32 Failure behavior

### Invalid room route

- show route recovery state;
- offer nearest valid room or home route;
- preserve shell.

### Missing room model

- use generic room presentation;
- keep device navigation functional.

### Failed default device

- remain in room overview;
- choose next available device only when configured.

## 3.2.35 Interaction contract

```ts
interface RoomNavigationController {
  preview(roomId: string): void;

```

## 3.2.36 Analytics and privacy

Room usage may be tracked locally for quick access and recency ranking.

Rules:

- no external analytics by default;
- room names and entity information never leave Home Assistant;
- local ranking can be disabled;
- ranking data is nonessential and safely resettable.

## 3.2.37 Acceptance criteria

Task 3.2 is approved when:

1. Every room is reachable from the navigation shell.
2. Room switching preserves one persistent application and WebGL stage.
3. Same-floor and cross-floor transitions are defined.
4. Default-device resolution is deterministic.
5. Room overview does not become a tile dashboard.
6. Empty, unassigned and unavailable room states are designed.
7. New rooms do not silently reorder existing navigation.
8. Previous/next behavior is deterministic.
9. Touch, pointer and keyboard navigation are supported.
10. Reduced-motion alternatives are specified.
11. Loading and failure states preserve continuity.
12. Room selection is deep-linkable and restorable.
13. User room order and overrides remain authoritative.
14. Device handoff begins only after room context resolves.
15. The design can proceed to the device configurator without redefining room behavior.

## 3.2.38 UX decisions

```text
UXD-001 Room routes change without remounting the stage
UXD-002 Room overview remains spatial and card-free
UXD-003 Default devices resolve through explicit priority
UXD-004 New rooms preserve existing order
UXD-005 Swipe room navigation is optional and off by default
UXD-006 Empty and unavailable rooms remain navigable
UXD-007 Cross-floor transitions use vertical spatial cues
UXD-008 User overrides remain authoritative
UXD-009 Preview never commits services or route state
UXD-010 Room transitions complete before device manipulation
```
