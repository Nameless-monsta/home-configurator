# 3.3 Device Configurator

**Status:** Complete  
**Phase:** 3 — UX and Visual Design

## Objective

Define the device-level experience that turns a selected Home Assistant device into one coherent, cinematic configuration surface rather than a stack of cards.

## Core composition

```text
Persistent navigation shell
+ Device identity
+ One dominant 3D object
+ Contextual capability controls
+ State readout
+ Previous/next device navigation
```

The 3D object remains the primary surface. Controls appear only when needed and never enclose the object in a conventional dashboard card.

## Device states

- `loading`
- `ready`
- `interactive`
- `pending`
- `confirmed`
- `unavailable`
- `partially_available`
- `misconfigured`
- `error`

## Entry behavior

On device selection:

1. Room context remains visible.
2. Current room object or previous device exits using the approved transition family.
3. Camera travels to the device framing preset.
4. Device identity and state readout enter.
5. Applicable controls resolve from capabilities.
6. Default control is selected only when configured or unambiguous.

## Information hierarchy

### Primary

- device name
- current state
- dominant control value
- active 3D object

### Secondary

- room name
- availability
- supporting attributes
- mode/source/preset

### Deferred

- diagnostics
- entity IDs
- integration metadata
- consumable details
- unsupported attributes

## Configurator modes

```text
Overview
Control
Details
Exploded
Diagnostics (developer/admin only)
```

Only one primary mode is active at a time.

## Control resolution

Controls are generated from normalized capabilities.

Examples:

```text
Power → toggle/power action
Brightness → continuous spatial control
Color → color sphere
Target temperature → temperature control
Volume → continuous media control
Cleaning → vacuum action set
```

Controls do not inspect raw Home Assistant domains or call services directly.

## Layout

### iPad landscape

- object occupies approximately 45–58% of viewport width;
- identity and state align left;
- configurator rail sits right or lower-right;
- previous/next navigation remains available;
- all primary controls fit without vertical scrolling.

### iPhone landscape

- one control family at a time;
- compact identity block;
- configurator becomes contextual overlay;
- object remains dominant.

### iPhone portrait

- object uses upper 55–62%;
- controls use a bottom dock;
- advanced details collapse;
- no card stack is introduced.

### Desktop

- object may offset from center to accommodate a wider control rail;
- hover can preview but never becomes required.

## Gesture ownership

```text
One-finger drag → rotate/manipulate device
Two-finger gesture → active continuous control where supported
Tap → select object part or action
Long press → optional detail affordance
Pinch → enabled only for explicit supported interactions
```

When a control captures a gesture, object rotation is suspended.

## Optimistic feedback

```text
Gesture
→ immediate visual preview
→ semantic command
→ pending indicator
→ Home Assistant confirmation
→ settle to confirmed state
```

Conflicting confirmation transitions smoothly to the authoritative value.

## Device switching

Previous/next device navigation:

- follows room-defined order;
- preserves room context;
- does not remount the WebGL canvas;
- preloads adjacent models where budget allows;
- retains per-device last control only when enabled.

## Unavailable and partial states

Unavailable:

- model remains visible;
- emission and active motion stop;
- material contrast reduces;
- unsupported actions disable;
- reason is displayed.

Partially available:

- available controls remain active;
- failed capability is disabled independently;
- the entire device is not blocked.

## Exploded mode

Exploded mode is available only when model metadata supports it.

- camera pulls back;
- parts separate using authored vectors;
- component labels appear after separation;
- selected parts may reveal supporting data;
- reassembly is fully reversible.

## Accessibility

- every gesture has a standard control alternative;
- continuous controls expose slider semantics;
- focus order follows identity → primary control → secondary controls → navigation;
- state changes are announced;
- unavailable reasons are readable without relying on color;
- canvas includes an accessible description.

## Acceptance criteria

- one visual device can compose multiple HA entities;
- controls resolve from capabilities;
- no raw entity IDs appear in normal runtime UI;
- one persistent renderer is retained across device switches;
- iPad landscape requires no page scrolling for primary control;
- optimistic and confirmed states are visually distinct;
- one failed capability does not disable the entire device;
- every gesture has an accessible fallback;
- the interface never degrades into stacked Lovelace cards.
