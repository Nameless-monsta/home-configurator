# 3.5 Climate Interaction

**Status:** Complete  
**Phase:** 3 — UX and Visual Design

## Objective

Define a cinematic, capability-driven climate experience that presents temperature, HVAC mode, fan behavior and airflow as one coherent interaction rather than separate cards.

## Core composition

```text
Climate device or abstract thermostat object
+ Current temperature
+ Target temperature
+ HVAC state
+ Contextual mode controls
+ Airflow visualization
```

## Supported capabilities

- power
- current temperature
- target temperature
- HVAC mode
- preset mode
- fan mode
- swing mode
- humidity readout
- auxiliary heat where supported

Only supported controls are displayed.

## Primary interaction

### Target temperature adjustment

Preferred interaction:

- one-finger vertical or circular drag on the active temperature control;
- immediate target-temperature preview;
- step behavior follows the entity capability metadata;
- final value commits on release;
- plus/minus and slider fallbacks remain available.

The interaction must never interfere with generic object rotation. Entering temperature-control mode explicitly captures the gesture.

## Temperature presentation

### Current temperature

- smaller supporting value;
- clearly labeled;
- never confused with target temperature.

### Target temperature

- dominant numeric value while adjusting;
- uses local unit from Home Assistant;
- shows one decimal only when the entity step requires it.

### Range mode

For heat/cool ranges:

```text
Lower target
Upper target
```

Use a dual-value interaction with explicit active handle and accessible numeric alternatives.

## HVAC state mapping

```text
Cooling → cool airflow visualization
Heating → warm airflow visualization
Dry → restrained moisture-reduction field
Fan only → neutral airflow
Auto → state-driven neutral/active response
Idle → minimal ambient motion
Off → airflow stops
```

Color remains local to the climate visualization and does not recolor navigation chrome.

## Airflow visualization

Airflow is semantic, not decorative.

It may use:

- sparse particles;
- flow-field ribbons;
- subtle distortion;
- directional volume cues;
- swing-driven direction changes.

Rules:

- effect intensity follows fan mode;
- direction follows swing state where available;
- reduced-motion mode substantially simplifies or freezes particles;
- low-performance tiers use a lighter fallback;
- airflow never obscures readouts or controls.

## Mode selector

HVAC modes appear as a restrained segmented or rail selector.

- only reported modes are listed;
- active mode is unambiguous;
- selecting a mode produces immediate pending feedback;
- mode changes may trigger an authored lighting/airflow transition;
- destructive or unusual modes may require confirmation only when justified.

## Fan and swing controls

### Fan mode

- discrete selector when modes are named;
- continuous control only when the integration exposes a numeric percentage;
- active airflow density updates in preview.

### Swing mode

- discrete selector;
- optional visual louvre or airflow-direction response;
- unsupported axes remain hidden.

## Presets

Presets such as eco, comfort, away or sleep appear only when exposed by the entity.

Presets are secondary controls and do not replace the primary target-temperature interaction.

## Optimistic behavior

```text
Adjustment
→ local numeric and airflow preview
→ semantic command
→ pending state
→ Home Assistant confirmation
→ settle or reconcile
```

Recommended command throttling:

```text
Continuous adjustment: 120–250 ms
Final commit: immediate on release
Mode changes: immediate single command
```

## Device states

- `off`
- `idle`
- `cooling`
- `heating`
- `drying`
- `fan_only`
- `auto`
- `pending`
- `unavailable`
- `error`

## Layout

### iPad landscape

- climate object remains central or slightly offset;
- target temperature and state identity align with the object;
- mode and fan controls occupy a compact contextual rail;
- all primary controls fit without scrolling.

### iPhone landscape

- target temperature remains dominant;
- mode controls collapse into one contextual panel;
- secondary humidity and diagnostics hide first.

### iPhone portrait

- object or airflow field occupies upper region;
- target temperature uses the bottom control dock;
- mode selector opens as a secondary sheet;
- no conventional thermostat card is introduced.

### Desktop

- pointer and keyboard interaction both supported;
- hover may preview labels but is never required.

## Accessibility

- temperature control exposes slider semantics;
- plus/minus alternatives are always available;
- current and target temperatures have distinct labels;
- mode changes are announced;
- color is not the only indication of heating/cooling;
- reduced-motion mode retains textual state;
- all touch targets meet minimum size.

## Unavailable and partial states

Unavailable:

- keep the device/model visible;
- stop airflow;
- disable commands;
- show last known values only when clearly marked stale.

Partial availability:

- preserve temperature adjustment if mode metadata is missing;
- disable only unsupported controls;
- never invent fan or swing capabilities.

## Error handling

If the target value is rejected:

- show restrained error feedback;
- transition back to confirmed target;
- preserve other state;
- expose the reason when Home Assistant provides one.

## Acceptance criteria

- current and target temperatures are never ambiguous;
- controls appear only for detected capabilities;
- gesture input and object manipulation do not conflict;
- target changes preview immediately and reconcile with HA;
- airflow reflects HVAC, fan and swing state semantically;
- all primary controls fit on iPad landscape without page scrolling;
- accessible numeric alternatives provide complete operation;
- unavailable sub-capabilities do not disable the entire climate device;
- reduced-motion and essential performance tiers remain fully usable.
