# 3.7 — Vacuum Interaction

**Status:** Complete  
**Depends on:** 3.2 Room Navigation, 3.3 Device Configurator

## Objective

Define a spatial control experience for robot vacuums that presents the device, cleaning state, battery, map, rooms, dock, and maintenance data as one coherent system.

## Experience principles

- The robot vacuum is represented as one 3D device, not a stack of entities.
- Primary cleaning actions remain immediately available.
- Room and zone cleaning is spatial where map data exists.
- Diagnostic entities remain secondary.
- The interaction remains useful when no map is exposed.
- Manufacturer-specific features extend the experience without altering core behavior.

## Supported capabilities

- Start
- Pause
- Stop
- Return to dock
- Locate
- Fan or cleaning mode
- Battery
- Current cleaning state
- Room cleaning
- Zone cleaning
- Map selection
- Mop mode and water level
- Dock actions
- Consumables and maintenance
- Error reporting

## Core states

```text
DOCKED
CHARGING
IDLE
CLEANING
PAUSED
RETURNING
ERROR
UNAVAILABLE
```

Optional extended states:

```text
MOPPING
EMPTYING
WASHING_MOP
DRYING_MOP
UPDATING_MAP
```

## Default composition

- The 3D vacuum occupies the hero position.
- A restrained dock model appears when relevant.
- Device status and battery appear as compact peripheral information.
- Primary control remains start/pause or return depending on state.
- Secondary controls appear through the configurator rail.

## Primary control behavior

### Docked or idle

Primary action:

```text
Start cleaning
```

Secondary actions:

```text
Room/zone selection
Cleaning mode
Locate
```

### Cleaning

Primary action:

```text
Pause
```

Secondary actions:

```text
Stop
Return to dock
Cleaning mode where supported
```

### Paused

Primary action:

```text
Resume
```

### Returning

Primary action:

```text
Stop return, if supported
```

### Error

Primary action:

```text
View error
```

The interface never presents start, pause, and resume as three equal permanent buttons.

## 3D object behavior

### Idle

- Very subtle lidar or indicator movement may continue.
- Device remains grounded with a soft contact shadow.

### Cleaning

- Wheels, brushes, and lidar may animate where the model supports them.
- Motion suggests activity without pretending to reproduce the live physical path.

### Returning

- Object orientation may align toward the dock in an authored animation.

### Charging

- Local indicator and battery response activate.
- No decorative energy effects beyond a restrained device-originated glow.

### Error

- Active movement stops.
- Error state is communicated through text, icon, and restrained material response.

## Map modes

### No map

The experience still provides:

- start/pause;
- return;
- cleaning mode;
- battery;
- maintenance;
- current state.

### Static image map

- Map opens in a contextual full-stage overlay.
- Room regions may be selected if mapping metadata exists.
- The map remains a control surface, not a permanent dashboard panel.

### Structured map

Supports:

- room polygons;
- named rooms;
- zones;
- dock position;
- current position where available;
- no-go zones as read-only context in v1.

### Map privacy

- Maps remain local.
- Exports and diagnostics omit map images and coordinates by default.

## Room cleaning

Room selection flow:

```text
Open cleaning target
→ show rooms
→ select one or more rooms
→ review selection
→ start
```

Rules:

- Home Assistant room names and vacuum map room names may differ.
- Mapping is stored explicitly once confirmed.
- Selection is visually clear without relying only on colour.
- Unsupported multi-room selection falls back to one room at a time.

## Zone cleaning

- Zone drawing is available only when the integration exposes a compatible command path.
- One-finger drag draws or adjusts the selection rectangle.
- Two-finger gesture manipulates the map viewport.
- Coordinates are transformed by the adapter, not the UI component.
- Invalid or unsupported zones are rejected before service dispatch.

## Cleaning mode

Modes may include:

```text
Quiet
Balanced
Turbo
Max
Custom
```

The selector displays only supported values and preserves manufacturer labels where useful.

Changing mode while cleaning is optimistic only when the integration supports live mode changes.

## Mop controls

Optional capabilities:

- mop intensity;
- water level;
- vacuum-and-mop mode;
- mop-only mode;
- mop washing;
- mop drying.

These appear only when detected.

## Dock controls

Possible dock actions:

- empty bin;
- wash mop;
- dry mop;
- clean dock;
- pause dock action.

Dock controls are grouped under one contextual section and never confused with robot navigation.

## Battery and consumables

### Battery

- Displayed as a supporting readout.
- Charging state is distinct from battery percentage.
- Low battery may influence action availability.

### Consumables

Examples:

- main brush;
- side brush;
- filter;
- sensors;
- mop;
- dust bag.

Consumables live in a maintenance mode, not the primary control view.

## Errors

Error presentation includes:

- concise user-facing description;
- affected component where known;
- recommended action where available;
- raw integration code only in advanced details.

An error never removes access to return, locate, or other safe supported actions unless the integration disables them.

## Roborock-class device composition

A Roborock system may compose:

```text
vacuum entity
+ battery sensor
+ map camera/image
+ fan speed selector
+ mop selector
+ consumable sensors
+ dock controls
```

The runtime resolves these into one `ResolvedDevice` and one 3D object.

## Responsive behavior

### iPad landscape

- Vacuum remains dominant.
- Primary control and state align to one side.
- Map opens as a wide contextual overlay.
- Maintenance remains a separate mode.

### iPhone landscape

- Compact hero presentation.
- Primary action fixed near the lower edge.
- Map mode uses most of the viewport.

### iPhone portrait

- Object occupies the upper half.
- Start/pause/return controls use the bottom dock.
- Room and map selection become full-screen states.

### Desktop

- Hover may preview room names and map regions.
- Pointer wheel zoom is limited to active map mode.

## Accessibility

- State and battery are announced clearly.
- Primary action uses explicit text.
- Map regions are also available as a selectable list.
- Zone drawing has numeric/list alternatives where practical.
- Gesture-only actions are prohibited.
- Error descriptions are screen-reader accessible.

## Failure behavior

- Failed commands restore confirmed state smoothly.
- Missing maps fall back to non-map control mode.
- One broken supporting sensor does not disable cleaning.
- Missing room mapping prompts review rather than guessing silently.
- Unavailable devices preserve configuration and model identity.

## Acceptance criteria

1. One vacuum system can compose multiple Home Assistant entities.
2. Start, pause, resume, return, and error states are unambiguous.
3. The experience works without a map.
4. Room and zone controls are spatial when supported.
5. Maintenance data remains secondary.
6. Manufacturer-specific dock and mop actions extend core controls cleanly.
7. Map data remains local and private.
8. All spatial selections have accessible alternatives.
9. Missing supporting entities do not disable the full device.
10. The 3D object visibly reflects cleaning, charging, returning, and error states.
