# 3.8 — Scene Interaction

**Status:** Complete  
**Depends on:** 3.1 Navigation Shell, 3.2 Room Navigation, 3.3 Device Configurator

## Objective

Define how scenes, scripts, and multi-device actions are previewed, activated, confirmed, reversed where possible, and represented spatially without becoming a grid of preset buttons.

## Experience principles

- A scene is a coordinated state change, not a generic button.
- The interface should preview intent before activation where practical.
- Scene activation must visibly affect the room and its devices.
- Scene presentation remains secondary to direct device control.
- Scripts and scenes share interaction patterns but retain different semantics.
- Unavailable devices do not make the entire scene visually fail.

## Supported action types

- Home Assistant scenes
- Scripts
- Automation triggers exposed as actions
- User-defined quick actions
- Room-wide presets
- System-wide presets
- Temporary presentation presets

## Scene identity

Each scene may define:

- name;
- icon;
- room or system scope;
- affected devices;
- preview state patches;
- estimated duration;
- reversibility metadata;
- availability requirements;
- confirmation policy.

## Core states

```text
AVAILABLE
PREVIEWING
PENDING
ACTIVE
PARTIALLY_APPLIED
FAILED
UNAVAILABLE
```

`ACTIVE` is used only where the state can be inferred reliably. A stateless script returns to `AVAILABLE` after confirmation.

## Scene entry points

Scenes may be reached from:

- room overview;
- quick access;
- system navigation;
- full navigation overlay;
- contextual device actions;
- optional recent/favourite actions.

Scenes must not permanently occupy the central device stage unless the scene itself is selected.

## Scene selector

The selector uses a restrained list, rail, or spatial arrangement depending on viewport.

Each item may show:

- scene name;
- scope;
- small status marker;
- affected-device count;
- optional short description.

The selector must not become a wall of decorative thumbnails.

## Preview behavior

Preview is optional and depends on available metadata.

### Full preview

When preview patches exist:

```text
select scene
→ apply temporary presentation preview
→ show affected devices
→ confirm or cancel
```

The preview changes only the rendered state. It does not call Home Assistant services.

### Partial preview

When only some affected states are known:

- known devices preview;
- unknown effects are labelled clearly;
- activation remains possible.

### No preview

For opaque scripts:

- show action description;
- show affected scope if known;
- require confirmation only when risk warrants it.

## Activation flow

```text
Scene selected
→ optional preview
→ activation intent
→ optimistic transition begins
→ semantic command dispatched
→ Home Assistant confirms state changes
→ scene settles
→ partial failures surfaced locally
```

The interface must not claim success merely because the service call returned if expected state confirmation is still pending.

## Confirmation policy

### No confirmation

Use for:

- reversible lighting scenes;
- routine media scenes;
- low-risk room presets.

### Lightweight confirmation

Use for:

- scenes affecting many rooms;
- scripts with unclear effects;
- actions that may interrupt media or cleaning.

### Strong confirmation

Use for:

- security-related actions;
- locks, alarms, doors, or destructive scripts;
- actions explicitly marked as high risk.

Confirmation policy is metadata-driven, not guessed from labels alone.

## Visual response

Scene activation should coordinate:

- device material changes;
- local lighting changes;
- room ambience;
- camera framing where appropriate;
- temporary shell feedback;
- affected-device indicators.

The transition director owns choreography. Individual devices do not start unrelated animations.

## Affected-device presentation

During preview or activation:

- affected devices may gain a subtle focus marker;
- unaffected devices recede slightly;
- failed or unavailable devices remain individually identifiable;
- no permanent connecting lines are introduced in the background.

## Scene scope

### Device scene

Affects one composed device or system.

### Room scene

Affects one room.

### Floor scene

Affects one floor.

### Whole-space scene

Affects the full installation.

Scope must be visible before activation.

## Quick actions

Quick actions are curated scene/script shortcuts.

Rules:

- maximum visible count is responsive;
- favourites and room relevance rank first;
- quick actions remain text-led and restrained;
- long-press opens details instead of activating;
- accidental activation risk is assessed per action.

## Active-state inference

A scene may be considered active when:

- all required state predicates match;
- no conflicting higher-priority scene is active;
- confirmation has settled.

If inference is unreliable, use `last activated` rather than `active`.

## Partial application

A scene is partially applied when:

- some devices confirm expected states;
- one or more devices fail, remain unavailable, or time out.

Presentation:

```text
Scene applied to 7 of 9 devices
```

The user may inspect failed devices without losing the current room context.

## Reversal and undo

Undo is available only when the system captured a safe pre-scene state and all required commands are reversible.

```text
Before state snapshot
→ activate scene
→ offer temporary Undo
→ dispatch restoration commands
```

Rules:

- undo window is short and explicit;
- high-risk actions do not expose casual undo unless verified safe;
- scripts with external side effects are not marked reversible;
- snapshot data remains local and temporary.

## Scene editing boundary

Runtime supports selection, preview, activation, status, and details.

The visual editor supports:

- ordering;
- naming;
- scope assignment;
- confirmation policy;
- preview patches;
- quick-access inclusion;
- affected-device metadata.

Creating full Home Assistant automation logic remains outside the runtime scope.

## Responsive behavior

### iPad landscape

- Scene rail or selector appears beside the room stage.
- Preview keeps the room and affected objects visible.
- Primary confirmation remains reachable without scrolling.

### iPhone landscape

- Scene selector becomes a compact overlay.
- Preview state occupies most of the stage.
- Details collapse behind one action.

### iPhone portrait

- Scene list may become full-screen.
- Preview and confirmation use a bottom dock.
- Affected-device details open separately.

### Desktop

- Hover may preview metadata only.
- Activation always requires click or keyboard confirmation.
- Keyboard focus order follows scene order.

## Accessibility

- Scene names and scope are announced.
- Preview is explicitly announced as not yet applied.
- Confirmation controls use clear action language.
- Active and partial states are not indicated by colour alone.
- Affected devices are available as a semantic list.
- Undo status is announced through a live region.

## Error behavior

- Failed activation preserves the pre-scene visual state where possible.
- Partial failure identifies affected devices.
- Unavailable scene entities remain visible but disabled.
- One failing device does not hide successful device changes.
- Timeouts resolve to confirmed Home Assistant states.

## Acceptance criteria

1. Scenes are presented as coordinated state changes, not generic cards.
2. Preview never dispatches Home Assistant services.
3. Scope is visible before activation.
4. Confirmation policy reflects action risk.
5. Success is reconciled against expected state where possible.
6. Partial application identifies failed or unavailable devices.
7. Undo appears only for safely reversible actions.
8. Scene transitions coordinate all affected devices through one director.
9. Quick actions remain curated and responsive.
10. All preview, activation, error, and undo states are accessible.
