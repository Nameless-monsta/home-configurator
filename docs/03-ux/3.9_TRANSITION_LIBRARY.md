# 3.9 — Transition Library

**Status:** Approved baseline

## Purpose

The transition library defines how Home Configurator moves between application states. A transition is not decoration; it communicates hierarchy, continuity, causality and system confidence.

All route, room, device, control, overlay, model and scene changes must use a registered transition family. Components may request a transition but may not invent their own timing or easing.

## Core rules

1. The persistent shell remains stable unless the shell itself is the transition target.
2. The central 3D stage is continuous across routes.
3. One primary transition owns the viewport at a time.
4. Secondary micro-transitions may run only when they do not compete with the primary transition.
5. Navigation state commits at a defined midpoint, never opportunistically.
6. Transitions must be interruptible and reversible where user input requires it.
7. Confirmed Home Assistant state always wins over optimistic visual state.
8. Reduced-motion mode preserves hierarchy without camera travel, large scaling or stagger.
9. Failed assets and unavailable devices resolve into stable fallbacks rather than blank frames.
10. Transition quality must degrade before input latency degrades.

## Transition director

```text
Request
→ validate source and target
→ select transition family
→ reserve transition lock
→ prepare assets and route data
→ leave phase
→ commit phase
→ enter phase
→ focus restoration
→ release transition lock
```

```ts
interface TransitionRequest {
  id: string;
  family: TransitionFamily;
  from: TransitionTarget;
  to: TransitionTarget;
  reason: "user" | "history" | "state" | "reconcile" | "system";
  direction?: "forward" | "back" | "next" | "previous" | "up" | "down";
  priority?: number;
}
```

## Transition families

### TR-01 Shell reveal

Used for initial application entry.

Sequence:

1. Canvas and environment establish immediately.
2. Hero object resolves from proxy to model.
3. Context text appears.
4. Persistent navigation enters last.

Nominal duration: `900–1300 ms`.

### TR-02 Overlay open/close

Used for navigation, quick access, information and system overlays.

Open:

- scene exposure lowers slightly;
- overlay surface expands or reveals from its anchor;
- rows enter with restrained stagger;
- focus transfers after the surface is stable.

Close reverses the sequence without replaying a full stagger.

Nominal duration: `280–460 ms`.

### TR-03 Home to room

The selected room becomes the new spatial context.

- current hero or home object shifts from overview framing;
- camera moves toward room framing;
- room title replaces installation title;
- room devices become available after the context commit;
- no page flash or hard cut.

Nominal duration: `700–1100 ms`.

### TR-04 Room to room — same floor

Uses horizontal or arc-like spatial continuity.

- outgoing context yields in the selected direction;
- camera follows a restrained lateral arc;
- environment and room lighting interpolate;
- incoming default device may appear after room identity is established.

Nominal duration: `650–1000 ms`.

### TR-05 Room to room — cross floor

Uses vertical hierarchy rather than an arbitrary sideways slide.

- context compresses slightly;
- camera performs a short vertical transfer;
- floor label passes through the commit point;
- next room resolves with new floor context.

Nominal duration: `800–1200 ms`.

### TR-06 Room to device

The device becomes the hero object without breaking context.

- room representation recedes or dissolves into supporting environment;
- selected device moves to hero framing;
- device identity and primary state appear;
- contextual control rail becomes available last.

Nominal duration: `600–950 ms`.

### TR-07 Device to device — same room

Preferred behavior is authored model transformation or coordinated swap.

- outgoing device begins departure;
- camera preserves room orientation;
- model swap occurs near the visual midpoint;
- incoming object arrives with matched scale and visual mass;
- control rail updates after model identity becomes clear.

Nominal duration: `520–850 ms`.

### TR-08 Device to device — different room

Composes room and device transition logic:

```text
device release
→ room context transfer
→ incoming device reveal
```

The sequence must feel continuous, not like two complete transitions chained together.

Nominal duration: `850–1350 ms`.

### TR-09 Device to control

Used when entering color, climate, media, vacuum or another focused control.

- device remains visible;
- camera may reframe to expose the control gesture zone;
- non-relevant controls reduce in emphasis;
- focused control emerges from the device state or rail.

Nominal duration: `320–600 ms`.

### TR-10 Control exit

- active preview settles or cancels;
- camera returns to device framing;
- control detail retracts;
- focus returns to the control trigger.

Nominal duration: `260–500 ms`.

### TR-11 Model variant change

Used for finish, colorway or alternate model presentation.

Preferred order:

1. Freeze uncontrolled object manipulation.
2. Preserve camera and scale.
3. Morph material values where geometry is shared.
4. Cross-dissolve or authored part swap where geometry differs.
5. Restore manipulation.

Nominal duration: `350–750 ms`.

### TR-12 Scene activation

A scene transition visualizes a coordinated state change without turning every device into a separate animation.

- scene intent appears;
- environment and affected hero representation begin preview;
- command is sent;
- confirmed states reconcile progressively;
- scene settles into confirmed atmosphere.

Nominal duration: `500–1600 ms`, depending on affected state confirmation.

### TR-13 Availability failure

Used when the selected device becomes unavailable.

- emission and active motion reduce;
- material moves toward unavailable treatment;
- control input disables gracefully;
- reason appears without replacing the entire layout.

Nominal duration: `240–500 ms`.

### TR-14 Recovery

- confirmed state restores first;
- model material and motion return;
- controls reactivate;
- status message clears last.

Nominal duration: `320–650 ms`.

### TR-15 Loading proxy to final model

- proxy establishes framing immediately;
- final model is prepared off-stage;
- geometry resolves with matched pivot, scale and orientation;
- proxy disappears only after the final frame is ready.

No blank canvas is permitted.

### TR-16 Orientation/profile change

Used for device rotation or responsive breakpoint change.

- preserve current route and selected object;
- recalculate layout and safe areas;
- reframe camera;
- move controls to the new composition;
- avoid replaying route-entry transitions.

Nominal duration: `300–650 ms`.

## Transition phases

Every primary transition exposes:

```text
PREPARE
LEAVE
COMMIT
ENTER
SETTLE
COMPLETE
```

### Commit phase

At `COMMIT`:

- route state changes;
- accessible route title updates;
- selected room/device/control IDs change;
- history entry is written;
- outgoing focus is released.

Visual departure may begin before commit and arrival may continue after it.

## Interruption policy

### Reversible

- overlay open/close;
- object drag-to-preview;
- next/previous room gesture before commit;
- model manipulation.

### Cancel and redirect

- device switch requested during device switch;
- room selection during a pending room transition;
- control change before previous control settles.

### Non-interruptible critical section

Only the short route/model commit section may be locked. It should normally remain below `160 ms`.

## Queue and priority

Priority order:

```text
system/error
> direct user selection
> history navigation
> confirmed state reconciliation
> automatic discovery/reordering
> ambient transitions
```

Repeated navigation requests collapse to the latest valid target.

## Timing tokens

```text
instant: 0–120 ms
micro: 120–220 ms
compact: 220–360 ms
standard: 360–650 ms
spatial: 650–1100 ms
cinematic: 1100–1600 ms
```

No transition should be extended merely to appear premium.

## Easing vocabulary

- **slow-out:** object and overlay arrival
- **fast-out/slow-in:** route departure
- **spring restrained:** controls, small indicators, object settle
- **linear:** progress, scrubbing and continuous confirmed values
- **symmetric ease:** reversible overlays and compact UI changes

Raw easing curves are implementation details owned by HCML.

## Reduced-motion variants

- camera travel becomes a short opacity/scale handoff;
- parallax is removed;
- stagger is removed;
- object morph uses a compact dissolve;
- cross-floor travel uses label and context replacement rather than vertical camera movement;
- duration is reduced but state hierarchy remains perceptible.

## Performance fallbacks

Tier B:

- reduce post-processing during transition;
- simplify environment interpolation;
- limit simultaneous material animation.

Tier C:

- no expensive dissolve shader;
- use matched opacity and transform handoff;
- no particles;
- preserve model identity and input responsiveness.

## Accessibility

- route changes announce the new room/device/control after commit;
- focus moves only after the target is interactive;
- canceled transitions restore previous focus;
- no transition is required to understand state;
- motion-sensitive users receive reduced-motion behavior automatically.

## Diagnostics

Developer mode exposes:

- active transition ID and family;
- source and target;
- phase;
- elapsed/expected duration;
- queue length;
- interruption reason;
- dropped frames;
- asset readiness;
- reduced-motion/performance override.

## Acceptance criteria

1. Every route and overlay change maps to a registered transition family.
2. One persistent WebGL stage is maintained.
3. Route commit timing is deterministic.
4. Navigation requests are interruptible without visual corruption.
5. Same-floor and cross-floor room changes are visually distinct.
6. Device changes preserve scale, orientation and spatial continuity.
7. Control entry keeps the device visible.
8. Loading never produces a blank stage.
9. Unavailable and recovery transitions are defined.
10. Scene activation reconciles optimistic and confirmed state.
11. Reduced-motion behavior exists for every primary family.
12. Tier B and C fallbacks preserve hierarchy and input latency.
13. Focus and announcements follow transition commit.
14. Diagnostics can identify transition ownership and failure.
