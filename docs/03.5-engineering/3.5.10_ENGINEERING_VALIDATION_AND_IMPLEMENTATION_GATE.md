# 3.5.10 — Engineering Validation and Implementation Gate

**Status:** Approved for Phase 4 implementation  
**Phase:** 3.5 — Engineering Specifications  
**Validated against:** 3.5.1 through 3.5.9

## 1. Purpose

This document is the final engineering review for the pre-development specification set. It determines whether a contributor can implement the first vertical slice without inventing ownership rules, runtime ordering, gesture behavior, Home Assistant reconciliation, rendering policy, performance limits, asset conventions, plugin contracts or schema behavior.

## 2. Final decision

**Phase 3.5 is approved. Phase 4 may begin.**

No unresolved architectural contradiction blocks implementation.

The approved implementation baseline is:

```text
Runtime execution model
→ Camera engine
→ Renderer/frame pipeline
→ Input arbitration
→ Home Assistant runtime
→ Performance manager
→ Asset pipeline
→ Plugin registries
→ Blueprint/runtime schema
```

The Phase 4 codebase must implement these contracts. It must not silently replace them with feature-local alternatives.

## 3. Validation method

The review checked each specification across the following dimensions:

1. ownership;
2. lifecycle;
3. deterministic ordering;
4. public interfaces;
5. failure isolation;
6. accessibility;
7. responsive behavior;
8. performance measurability;
9. extension safety;
10. testability;
11. implementation readiness.

## 5. Runtime-order validation

The required input-to-frame sequence is deterministic:

```text
Browser input
→ input normalization
→ gesture arbitration
→ semantic intent
→ runtime validation
→ optimistic state
→ command queue
→ HA adapter dispatch
→ confirmed-state reconciliation
→ immutable frame snapshot
→ camera/transition/animation update
→ render
→ diagnostics
```

Critical rules confirmed:

- Home Assistant service calls are not issued by UI components;
- the renderer does not read mutable integration objects directly;
- route commits occur at declared transition commit points;
- confirmed Home Assistant state remains authoritative;
- user input is not delayed by visual quality work;
- stale or failed commands reconcile locally rather than resetting the application;
- one persistent WebGL stage remains mounted across navigation.

**Result:** Pass.

## 6. Camera and renderer compatibility

Validated:

- camera mutations occur only through Camera Engine plans;
- camera presets consume model bounds, anchors and Layout Engine safe areas;
- renderer applies camera poses but does not author camera behavior;
- orientation and breakpoint changes reframe without replaying route entry;
- proxy assets preserve a valid frame until final models are ready;
- rendering quality may degrade without changing semantic state;
- context loss and restoration have defined recovery behavior;
- scene resources have explicit creation and disposal ownership.

**Result:** Pass.

## 7. Input and gesture conflict validation

The arbitration contract resolves the major touch conflicts:

- object rotation versus active continuous control;
- one-finger sphere rotation versus two-finger brightness;
- climate adjustment versus object manipulation;
- volume gesture versus device orbit;
- map selection versus map viewport manipulation;
- edge-swipe room navigation versus device/control gestures;
- browser scroll and zoom versus full-screen interaction.

Locked rules:

- a gesture has one owner at a time;
- explicit control mode outranks generic object manipulation;
- pointer-count changes trigger defined re-arbitration or cancellation;
- gesture capture is released on completion, cancellation, route change and error;
- all gesture-only interactions have keyboard or standard-control alternatives;
- sphere geometry never changes size to represent brightness.

**Result:** Pass.

## 8. Home Assistant runtime validation

Validated:

- connection lifecycle includes connect, subscribe, reconnect, stale and dispose states;
- discovery and entity resolution are separate from presentation;
- raw Home Assistant entities normalize into canonical device/capability state;
- commands use semantic operations before adapter translation;
- continuous commands are throttled and coalesced;
- final release commits the precise value;
- confirmed state reconciles optimistic state;
- partial capability failure does not disable the complete device;
- unavailable state preserves route, identity and last safe presentation;
- reconnect performs controlled resubscription and state refresh;
- local configuration and private data remain local by default.

**Result:** Pass.

## 9. Performance and resource validation

The specifications provide measurable budgets for:

- frame time;
- FPS targets;
- draw calls;
- triangles;
- active lights and shadows;
- texture dimensions and memory;
- model and bundle size;
- particles and post-processing;
- command/update frequency;
- CPU and GPU diagnostics;
- background and hidden-tab behavior.

The Full, Balanced and Essential tiers preserve the same functional meaning.

Required degradation order:

```text
nonessential post-processing
→ particle density
→ shadow quality
→ environment resolution
→ material complexity
→ LOD/model detail
```

Input responsiveness, semantic controls and readable state may not be degraded.

**Result:** Pass, with real-device thresholds to be confirmed during the first vertical prototype.

## 10. Asset-pipeline validation

Validated:

- GLB/glTF is the canonical runtime format;
- units, axes, pivots, bounds, names and anchors are defined;
- compression, texture and material rules are versioned;
- LOD generation and validation are deterministic;
- model manifests expose capabilities, variants, animation clips and camera metadata;
- invalid assets fail before registry admission;
- exact-model failure resolves to profile/category/generic fallback;
- licensing and attribution metadata are required;
- runtime never scans arbitrary folders for models.

**Result:** Pass.

## 11. Plugin and extension validation

Validated:

- plugins register through typed registries;
- lifecycle includes install, validate, load, initialize, register, ready, unload and quarantine;
- permissions are declared and granted explicitly;
- plugins cannot mutate core state or Three.js internals directly;
- semantic commands remain subject to core validation;
- namespaces prevent identifier collisions;
- dependencies and compatibility ranges are checked before activation;
- extension failure is isolated and diagnosable;
- safe mode can disable third-party packages;
- updates support rollback.

**Result:** Pass.

## 12. Blueprint and runtime-schema validation

Validated:

- portable blueprints contain intent rather than user entity IDs;
- runtime mappings are local and installation-specific;
- placeholders declare profile, role and required capabilities;
- mapping confidence and evidence are retained;
- user overrides always win;
- import supports validation, preview, conflict review and rollback;
- migrations are deterministic and versioned;
- unknown extension namespaces are preserved or rejected according to declared policy;
- portable exports scrub entity IDs, room names, coordinates and sensitive integration metadata;
- unresolved optional mappings degrade gracefully;
- unresolved required mappings block only affected functionality.

**Result:** Pass.

## 13. Accessibility and reduced-motion validation

Across the specifications:

- every gesture has a non-gesture alternative;
- focus ownership is explicit during navigation and transitions;
- route and state changes have announcement points;
- reduced motion preserves hierarchy and state meaning;
- colour and animation are never the only communication channels;
- the canvas has an accessible semantic counterpart;
- VoiceOver and keyboard paths are included in the first prototype gate.

**Result:** Pass for implementation; device testing remains mandatory.

## 14. Failure-isolation validation

Stable fallback behavior exists for:

- invalid routes;
- missing rooms and devices;
- unavailable entities;
- partial capability loss;
- service-call rejection;
- timeout and reconnect;
- missing model assets;
- invalid model metadata;
- WebGL context loss;
- performance pressure;
- plugin initialization failure;
- incompatible package versions;
- blueprint validation and migration failure.

No listed failure requires a blank page or full application reset.

**Result:** Pass.

## 15. First vertical slice

Phase 4 must begin with one narrow end-to-end implementation rather than broad device coverage.

### Required scope

```text
Application shell
+ one persistent WebGL canvas
+ one room
+ one color-capable light
+ one generic or exact lamp model
+ room-to-device transition
+ power control
+ color sphere
+ two-finger brightness gesture
+ HA connection and live subscription
+ optimistic/confirmed reconciliation
+ responsive iPad and iPhone layout
+ reduced-motion variant
+ basic keyboard and VoiceOver path
+ diagnostics overlay
```

### Explicitly excluded from the first slice

- visual editor;
- marketplace;
- broad manufacturer coverage;
- vacuum maps;
- full media remote;
- scene authoring;
- complex plugin distribution;
- polished model pack library.

These remain in architecture but must not dilute the first validation target.

## 16. Mandatory implementation tests

Before expanding the prototype, the following must pass:

1. application boots with no Home Assistant connection and shows a stable fallback;
2. one WebGL canvas survives room and device route changes;
3. direct manipulation remains responsive during state updates;
4. sphere rotation resolves hue and saturation accurately;
5. two-finger vertical movement changes brightness without scaling the sphere;
6. command throttling reduces service-call volume during continuous input;
7. final pointer release sends the exact final value;
8. rejected commands reconcile to confirmed state without route loss;
9. disconnect preserves last confirmed presentation and disables unsafe commands;
10. reconnect resubscribes without duplicate listeners;
11. iPad landscape sustains the accepted frame budget under the prototype scene;
12. quality downgrade preserves controls and state meaning;
13. model failure produces a generic fallback with the same capabilities;
14. reduced-motion mode removes camera travel and ambient loops;
15. keyboard and VoiceOver can operate power, hue, saturation and brightness;
16. context loss restores a usable scene or presents a controlled fallback;
17. all resources dispose cleanly after repeated device switching;
18. diagnostics identify active gesture, command, transition, quality tier and frame cost.

## 17. Remaining non-blocking decisions

The following are intentionally deferred and do not block Phase 4:

- exact monorepo tooling and package manager;
- final public project license;
- final shader implementation choices;
- exact initial lamp model;
- final CI device matrix;
- distribution signing policy;
- marketplace UI;
- long-term documentation-site framework.

They must be decided before their relevant release gate.

## 18. Change-control rule

After this approval:

- implementation may refine constants and measured thresholds;
- implementation may not change ownership or lifecycle semantics silently;
- any contradiction discovered in code must produce an amendment to the relevant 3.5 document;
- deviations must be recorded through an ADR;
- the first vertical slice is the source of measured values, not a reason to bypass contracts.
