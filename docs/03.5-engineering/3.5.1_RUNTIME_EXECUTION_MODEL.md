# 3.5.1 — Runtime Execution Model

**Status:** Approved baseline  
**Depends on:** Phase 2 architecture, Phase 3 UX, transition and animation libraries

## Objective

Define the deterministic runtime lifecycle that coordinates Home Assistant state, application state, commands, transitions, animations, rendering, accessibility, diagnostics, and user input.

The runtime must preserve one persistent application shell and one persistent rendering stage while preventing subsystems from mutating each other directly.

## Runtime owners

```text
Application Runtime
├── State Store
├── Event Bus
├── Command Coordinator
├── Transition Director
├── Animation Scheduler
├── Input Router
├── Scene/Render Coordinator
├── Home Assistant Adapter
├── Package Registry
├── Performance Manager
└── Diagnostics Service
```

Each owner has one responsibility and one public boundary.

## Non-negotiable rules

1. Home Assistant adapters never manipulate UI or Three.js objects directly.
2. Render components never call Home Assistant services directly.
3. Input handlers emit semantic intents rather than domain-specific service calls.
4. Commands pass through the command coordinator.
5. Route and hero-object movement pass through the transition director.
6. In-state motion passes through the animation scheduler.
7. Confirmed backend state is authoritative.
8. Optimistic state is explicit, temporary, and reversible.
9. One primary transition owns the viewport at a time.
10. The rendering loop may read state snapshots but may not become the canonical state store.

## Runtime lifecycle

```text
BOOTSTRAP
→ CONNECTING
→ HYDRATING
→ RESOLVING
→ READY
→ ACTIVE
↔ DEGRADED
↔ RECONNECTING
→ DISPOSING
```

### BOOTSTRAP

- load static configuration;
- initialize package registry;
- validate schema versions;
- create state store, event bus and diagnostics;
- mount persistent shell and rendering stage;
- establish proxy scene before remote data is available.

### CONNECTING

- establish Home Assistant connection;
- negotiate API/runtime compatibility;
- register subscriptions;
- expose connection state without blocking navigation shell rendering.

### HYDRATING

- ingest entity, device, area and floor registries;
- restore safe local preferences and last valid route;
- hydrate cached confirmed state where available;
- mark cached values as stale until live confirmation.

### RESOLVING

- run entity abstraction;
- resolve rooms and composed devices;
- detect capabilities;
- resolve model, blueprint and package dependencies;
- build runtime scene graph references.

### READY

- first valid route and presentation are available;
- critical assets are loaded or represented by proxies;
- input becomes available;
- route title and accessibility context are announced.

### ACTIVE

Normal operation: input, commands, transitions, animations and state reconciliation.

### DEGRADED

Used when one or more nonfatal subsystems fail or quality must reduce.

Examples:

- optional model unavailable;
- supporting entity unavailable;
- performance tier lowered;
- optional plugin failed;
- stale cached state is displayed.

### RECONNECTING

- retain last confirmed presentation;
- freeze or queue commands according to policy;
- continue local navigation;
- reconnect subscriptions;
- rehydrate authoritative state;
- reconcile without replaying full startup transitions.

### DISPOSING

- stop loops;
- cancel subscriptions;
- settle or cancel pending commands;
- release GPU and DOM resources;
- persist safe local preferences;
- dispose plugins in reverse registration order.

## State layers

```text
Confirmed State
Optimistic State
Presentation State
Transient Interaction State
Derived State
```

### Confirmed state

The latest accepted state from Home Assistant or another authoritative adapter.

### Optimistic state

A temporary prediction associated with a command ID. It must contain:

- source command;
- expected result;
- creation timestamp;
- timeout;
- rollback value;
- reconciliation policy.

### Presentation state

Current route, selected room/device/control, camera preset, overlay state, transition phase, quality tier and active visual modifiers.

### Transient interaction state

Pointer capture, gesture owner, drag deltas, velocity, focused control and preview values. It is never persisted as installation configuration.

### Derived state

Computed selectors such as room summaries, primary actions, current media owner and resolved availability. Derived state is recomputable and not independently mutated.

## Intent pipeline

```text
Raw input
→ Input Router
→ Semantic Intent
→ Policy validation
→ State/transition/command request
→ Immediate presentation feedback
→ Backend dispatch where required
→ Confirmation or failure
→ Reconciliation
```

Example:

```text
Two-finger vertical movement on color control
→ AdjustBrightnessIntent
→ capability and ownership validation
→ optimistic brightness preview
→ coalesced SetBrightnessCommand
→ Home Assistant service call
→ state subscription confirmation
→ settle or rollback
```

## Frame lifecycle

The renderer uses one animation-frame loop.

```text
1. Read time and visibility
2. Consume input sample
3. Advance gesture session
4. Advance active transition
5. Advance registered animations
6. Read immutable presentation snapshot
7. Update scene graph transforms/materials
8. Apply camera state
9. Apply performance decisions
10. Render
11. Record diagnostics
12. Publish post-frame events if needed
```

Rules:

- Home Assistant network events do not mutate the scene mid-frame;
- incoming events update the store before the next frame snapshot;
- command dispatch is not tied to frame rate;
- high-frequency service calls are throttled/coalesced independently;
- hidden tabs pause nonessential rendering and animation.

## Store transaction ordering

A store transaction follows:

```text
validate
→ reduce canonical state
→ recompute affected selectors
→ notify priority subscribers
→ schedule presentation work
→ emit diagnostic event
```

Priority subscribers:

1. safety and connection state;
2. command reconciliation;
3. route and transition ownership;
4. renderer presentation;
5. UI readouts;
6. diagnostics and nonessential observers.

## Event bus

Events describe completed facts, not requests for mutation.

Examples:

```text
RuntimeReady
ConnectionChanged
DiscoveryResolved
RouteCommitted
TransitionCompleted
CommandDispatched
CommandConfirmed
CommandFailed
ModelReady
QualityTierChanged
```

Requests use explicit controller methods or intent dispatch. This prevents circular event-driven mutation.

## Command lifecycle

```text
CREATED
→ VALIDATED
→ OPTIMISTIC
→ QUEUED
→ DISPATCHED
→ AWAITING_CONFIRMATION
→ CONFIRMED | FAILED | TIMED_OUT | CANCELED
→ SETTLED
```

Command requirements:

- unique ID;
- target capability;
- desired value/action;
- source interaction;
- coalescing key where applicable;
- timeout and retry policy;
- expected confirmation predicate;
- rollback or reconcile behavior.

Continuous controls may replace queued commands with the latest valid value, but the release/final command is never discarded.

## Transition and animation ownership

The transition director owns route, room, device, control and model identity changes.

The animation scheduler owns motion inside the current state.

```text
Transition owns: departure, commit, arrival, focus transfer
Animation owns: idle, state response, manipulation settle, pending, confirmation
```

When a primary transition begins:

- conflicting ambient animations pause;
- direct manipulation releases or cancels safely;
- capability previews settle or roll back according to policy;
- the transition receives exclusive camera ownership.

## Focus and accessibility lifecycle

```text
Intent begins
→ preserve source focus reference
→ transition/control state changes
→ route commits and accessible title updates
→ target becomes interactive
→ focus moves to target or restored logical control
→ live-state message announces result when needed
```

Animation completion is not normally required before keyboard use resumes. Focus waits only for state integrity and target availability.

## Error boundaries

Each subsystem must fail locally where possible.

```text
Plugin failure → disable plugin, preserve core
Model failure → proxy/fallback model
Supporting entity failure → partial capability availability
Command failure → affected control rollback
Renderer effect failure → lower quality/fallback material
Connection failure → reconnecting state with last confirmed view
```

Fatal runtime failure is reserved for corrupted core configuration or inability to create any safe application state.

## Persistence rules

May persist locally:

- user preferences;
- room/device order overrides;
- last safe route;
- quality preference;
- accepted mappings;
- blueprint configuration;
- model assignments.

Must not persist as authoritative runtime state:

- optimistic values;
- pending commands;
- pointer/gesture sessions;
- transient transition phase;
- unconfirmed device state.

## Diagnostics contract

The diagnostics service records:

- runtime phase;
- connection state;
- store transaction count and duration;
- input owner;
- active command count;
- command latency and failure;
- active transition and animation IDs;
- frame time/FPS;
- memory and GPU estimates where available;
- package/plugin failures;
- stale state indicators.

Private entity values, room names and map data are excluded from exported diagnostics by default.

## Public runtime interface

```ts
interface HomeConfiguratorRuntime {
  start(): Promise<void>;
  dispose(): Promise<void>;
  dispatch(intent: RuntimeIntent): Promise<IntentResult>;
  navigate(target: NavigationTarget): Promise<void>;
  execute(command: SemanticCommand): Promise<CommandResult>;
  getSnapshot(): RuntimeSnapshot;
  subscribe(listener: RuntimeListener): () => void;
  getDiagnostics(): RuntimeDiagnostics;
}
```

Subsystems receive narrower interfaces and must not depend on the complete runtime object.

## Acceptance criteria

1. All runtime owners and mutation boundaries are explicit.
2. One deterministic input-to-command pipeline exists.
3. Confirmed, optimistic, presentation and transient states are separated.
4. The frame lifecycle has a fixed order.
5. Network events never mutate renderer objects directly.
6. Commands expose validation, dispatch, confirmation, failure and settlement phases.
7. Transition and animation ownership cannot conflict.
8. Reconnection preserves the last valid presentation and rehydrates safely.
9. Failures remain local whenever a stable fallback exists.
10. Focus and accessibility updates follow route/control commit rules.
11. Diagnostics identify ownership, latency and runtime phase.
12. The runtime can support the first vertical prototype without inventing new lifecycle rules.
