# 3.5.2 — Camera Engine Contract

**Status:** Approved engineering baseline  
**Phase:** 3.5 — Engineering Specifications  
**Depends on:** 3.5.1 Runtime Execution Model, 3.1 Navigation Shell, 3.2 Room Navigation, 3.3 Device Configurator, 3.9 Transition Library

## 1. Purpose

The Camera Engine owns every runtime camera pose, framing change, responsive reframe and camera transition in Home Configurator.

No room module, device controller, plugin, model, control or visual component may mutate the active camera directly.

The engine exists to guarantee:

- one persistent camera system across the application;
- deterministic framing for rooms, devices and controls;
- responsive compositions that preserve the approved UX hierarchy;
- safe interruption and redirection during navigation;
- stable object manipulation without camera conflict;
- reduced-motion alternatives;
- measurable performance and diagnostics;
- graceful behavior when model metadata is incomplete.

## 2. Ownership boundary

```text
Route / room / device / control intent
→ Transition Director
→ Camera Engine request
→ Camera plan
→ Renderer applies pose
```

The Camera Engine owns:

- active camera identity;
- camera position and orientation;
- projection parameters;
- framing target and safe framing region;
- camera presets;
- transition paths;
- responsive reframing;
- user inspection offsets;
- interruption and redirection;
- camera diagnostics.

The Camera Engine does not own:

- route state;
- selected room or device state;
- object transforms caused by device manipulation;
- UI layout;
- Home Assistant state;
- lighting;
- transition sequencing outside the camera portion;
- model asset loading.

## 4. Camera modes

```text
BOOT
HOME_OVERVIEW
ROOM_OVERVIEW
DEVICE_HERO
CONTROL_FOCUS
MODEL_INSPECTION
EXPLODED_VIEW
MAP_FOCUS
OVERLAY_BACKGROUND
TRANSITIONING
RECOVERY
```

Only one primary mode is active at a time.

### BOOT

- establishes a valid camera immediately;
- uses a generic installation framing if scene metadata is not yet available;
- never waits for final model assets before producing a stable frame.

### HOME_OVERVIEW

- frames the installation, floor plan or abstract home representation;
- preserves navigation-safe margins;
- avoids excessive perspective distortion on iPad.

### ROOM_OVERVIEW

- frames the room-level composition;
- reserves room identity and quick-access areas;
- establishes the room coordinate context used by subsequent device framing.

### DEVICE_HERO

- frames one dominant device or composed system;
- uses model bounds, authored anchor and visual-mass correction;
- keeps the object large without colliding with text or controls.

### CONTROL_FOCUS

- reframes the device to make room for an active control;
- retains the device as a visible subject;
- never replaces the device with a control-only page.

### MODEL_INSPECTION

- allows controlled orbit, limited zoom and optional pan;
- remains constrained by authored limits;
- returns predictably to the previous hero pose.

### EXPLODED_VIEW

- expands framing to include all authored parts and labels;
- preserves legibility of the complete object;
- disables unsafe zoom limits while parts are separated.

### MAP_FOCUS

- used for vacuum maps and other spatial control surfaces;
- applies orthographic or low-distortion perspective framing where configured;
- keeps map manipulation independent from the main hero-camera orbit.

### OVERLAY_BACKGROUND

- preserves the current scene beneath a navigation or details overlay;
- may reduce parallax or apply a minor authored offset;
- never performs decorative roaming behind an open overlay.

## 5. Projection policy

The default runtime camera is perspective.

Orthographic projection is allowed only for explicitly declared cases, such as:

- vacuum maps;
- plan views;
- technical inspection modes;
- accessibility or editor views that require low distortion.

Projection changes must be planned transitions. Components may not toggle projection directly.

```ts
interface CameraProjection {
  type: "perspective" | "orthographic";
  fov?: number;
  near: number;
  far: number;
  zoom?: number;
}
```

Baseline perspective guidance:

```text
Hero/object views: 28–42° vertical FOV
Room/home overview: 32–50° vertical FOV
Close inspection: 24–38° vertical FOV
```

Final values are preset-specific and must pass device testing.

## 6. Camera preset contract

Every meaningful view resolves to a registered preset.

```ts
interface CameraPreset {
  id: string;
  mode: CameraMode;
  projection: CameraProjection;
  target: CameraTargetReference;
  framing: FramingPolicy;
  orientation?: CameraOrientationHint;
  safeAreaPolicy: string;
  interactionPolicy: CameraInteractionPolicy;
  responsiveVariants?: Record<string, Partial<CameraPreset>>;
  reducedMotionVariant?: Partial<CameraPreset>;
  transitionFamily?: string;
  version: number;
}
```

Preset sources, in priority order:

```text
1. Explicit user override
2. Blueprint override
3. Exact model preset
4. Device-profile preset
5. Category preset
6. Generic fallback preset
```

A missing exact model preset must never block rendering.

## 7. Target references

The camera never targets arbitrary world coordinates supplied by UI components.

Valid target references:

```ts
type CameraTargetReference =
  | { kind: "scene-node"; nodeId: string }
  | { kind: "model-anchor"; modelId: string; anchor: string }
  | { kind: "bounds"; nodeIds: string[] }
  | { kind: "room"; roomId: string }
  | { kind: "map"; mapId: string }
  | { kind: "fixed"; point: Vec3 };
```

Fixed points are restricted to core/editor-authored presets.

## 8. Model camera metadata

Models may declare:

```ts
interface ModelCameraMetadata {
  heroAnchor?: Vec3;
  lookAtAnchor?: Vec3;
  upAxis?: Vec3;
  preferredYaw?: number;
  preferredPitch?: number;
  minDistance?: number;
  maxDistance?: number;
  inspectionBounds?: CameraInspectionBounds;
  visualMassOffset?: Vec3;
  safePadding?: number;
}
```

When metadata is missing, the Framing Resolver derives a stable result from validated bounds and pivot data.

## 9. Framing resolver

The Framing Resolver computes a pose from:

- target world bounds;
- projection;
- viewport dimensions;
- safe areas;
- UI occupancy;
- authored padding;
- model visual-mass offset;
- active control region;
- device orientation;
- quality and accessibility mode.

```text
Target bounds
+ Safe frame
+ Projection
+ Authored hints
→ Required camera distance
→ Position/orientation solution
→ Constraint validation
```

The resolver must fit the complete required bounds without clipping while preserving the intended object scale.

## 10. Safe-area contract

The Camera Engine receives a normalized safe-frame description from the Layout Engine.

```ts
interface CameraSafeFrame {
  viewportWidth: number;
  viewportHeight: number;
  top: number;
  right: number;
  bottom: number;
  left: number;
  subjectRegion?: Rect;
  reservedRegions?: Rect[];
  devicePixelRatio: number;
}
```

Safe areas include:

- iOS status and home-indicator insets;
- persistent navigation shell;
- control rail or bottom dock;
- device identity/readout region;
- temporary overlay occupancy;
- keyboard and accessibility UI where reported.

The camera uses these regions for framing only. It does not position HTML UI.

## 11. Responsive profiles

Canonical profiles:

```text
IPAD_LANDSCAPE
IPAD_PORTRAIT
PHONE_LANDSCAPE
PHONE_PORTRAIT
DESKTOP_STANDARD
DESKTOP_WIDE
EMBEDDED_COMPACT
```

### iPad landscape

- canonical design target;
- object framing favors horizontal composition;
- control rails may reserve either side;
- camera movement remains restrained and cinematic;
- primary interactions require no page scrolling.

### Phone portrait

- hero object moves to the upper composition;
- bottom-control safe area is reserved;
- simultaneous depth effects are reduced;
- inspection zoom limits prevent the object from leaving the useful upper region.

### Orientation changes

Orientation change behavior:

```text
freeze current semantic target
→ recompute safe frame
→ resolve new responsive preset
→ reframe without route transition
→ restore interaction
```

The engine must not replay room or device entry animations after rotation.

## 12. Camera request contract

```ts
interface CameraRequest {
  id: string;
  presetId?: string;
  mode?: CameraMode;
  target: CameraTargetReference;
  reason:
    | "bootstrap"
    | "route"
    | "room"
    | "device"
    | "control"
    | "inspection"
    | "responsive"
    | "recovery"
    | "system";
  transitionFamily?: string;
  direction?: "forward" | "back" | "left" | "right" | "up" | "down";
  preserveInspectionOffset?: boolean;
  priority?: number;
}
```

Requests are submitted through the Transition Director for route-related movement. Direct camera requests are allowed only for inspection, responsive reframe, diagnostics and recovery.

## 13. Camera plan

Every accepted request resolves to an immutable plan.

```ts
interface CameraPlan {
  requestId: string;
  from: CameraPose;
  to: CameraPose;
  path: CameraPath;
  projectionFrom: CameraProjection;
  projectionTo: CameraProjection;
  durationMs: number;
  easingToken: string;
  commitProgress: number;
  interruptPolicy: CameraInterruptPolicy;
  reducedMotion: boolean;
}
```

The plan is computed before movement begins.

## 14. Camera pose

```ts
interface CameraPose {
  position: Vec3;
  quaternion: Quat;
  target: Vec3;
  projection: CameraProjection;
  focusDistance?: number;
}
```

Euler angles are not part of the public runtime contract.

## 15. Path families

Registered camera paths:

```text
DIRECT
LATERAL_ARC
DEPTH_APPROACH
DEPTH_RETREAT
VERTICAL_TRANSFER
ORBIT_REFRAme
MATCHED_HANDOFF
STATIC_REFRAme
```

### DIRECT

Used for compact corrections and reduced-motion variants.

### LATERAL_ARC

Used primarily for same-floor room movement.

### DEPTH_APPROACH / RETREAT

Used for home-to-room, room-to-device and device-to-room hierarchy.

### VERTICAL_TRANSFER

Used for cross-floor movement.

### ORBIT_REFRAME

Used when a device remains selected but the active control needs a new angle.

### MATCHED_HANDOFF

Used for device-to-device swaps where scale and visual mass must align.

### STATIC_REFRAME

Used for responsive layout changes and safe-area corrections.

Plugins may request registered path families but may not supply arbitrary executable path functions.

## 16. Interpolation rules

- position and orientation interpolate independently but remain synchronized by plan progress;
- quaternion slerp is required for rotation;
- target interpolation must not produce camera roll;
- FOV changes are limited and should not simulate zoom when a positional move is more appropriate;
- near/far planes may update discretely only when no clipping or depth artifact occurs;
- user-visible movement must be frame-rate independent;
- route camera movement uses Transition Library timing tokens.

## 17. Camera roll and horizon

Runtime camera roll is disabled by default.

Roll is allowed only for an explicitly authored effect with:

- bounded amplitude;
- reduced-motion fallback;
- no conflict with accessibility or control precision;
- explicit approval in a transition preset.

No normal room, device or control transition uses decorative roll.

## 18. Inspection controls

Inspection mode supports:

```text
One-finger drag / pointer drag → orbit
Pinch / wheel in focused mode → zoom
Optional two-finger pan → constrained target offset
Reset action → return to authored hero pose
```

Inspection constraints:

```ts
interface CameraInspectionBounds {
  minYaw: number;
  maxYaw: number;
  minPitch: number;
  maxPitch: number;
  minDistance: number;
  maxDistance: number;
  maxTargetOffset?: Vec3;
}
```

Rules:

- camera manipulation is enabled only in an explicit inspection-capable state;
- object control gestures take priority when a capability control owns the same pointer sequence;
- inspection offsets are transient by default;
- route changes discard inspection offsets unless the target explicitly preserves them;
- reset must be available without gesture input.

## 19. Zoom policy

Zoom changes camera distance by default, not object scale.

Orthographic map views may use projection zoom.

The engine must prevent:

- clipping through the object;
- entering model interiors unless inspection metadata permits it;
- zooming so far out that the selected device loses identity;
- zoom behavior while a two-finger capability gesture owns the pointers.

## 20. Collision and clipping guard

The Collision Guard protects against:

- camera entering target bounds;
- near-plane intersection;
- floor or environment penetration;
- invalid poses from corrupt metadata;
- controls or labels forcing impossible framing.

When a requested pose is invalid:

```text
validate authored pose
→ attempt constrained correction
→ resolve generic fallback pose
→ emit diagnostic warning
```

Camera failure must never create a blank or inverted frame.

## 21. Transition ownership and interruption

The Transition Director owns route-level camera timing.

The Camera Engine owns camera interpolation within that transition.

Interruption classes:

### Reversible

- overlay background offset;
- inspection orbit;
- pre-commit navigation preview.

### Redirectable

- room or device navigation before the commit phase;
- responsive reframe during a noncritical transition;
- user selecting a newer target.

### Locked critical section

Only the short commit segment may reject a new route camera request. Target duration remains below `160 ms`.

Repeated requests collapse to the latest valid semantic target.

## 22. Direct manipulation arbitration

The Camera Engine does not listen to raw pointers.

```text
Input Engine
→ Gesture arbitration
→ semantic camera intent
→ Inspection Controller
```

Camera intent examples:

```ts
type CameraIntent =
  | { type: "orbit"; deltaX: number; deltaY: number }
  | { type: "zoom"; delta: number }
  | { type: "pan"; deltaX: number; deltaY: number }
  | { type: "reset" };
```

This prevents the camera from competing with:

- color-sphere rotation;
- two-finger brightness control;
- climate adjustment;
- media volume control;
- vacuum map manipulation;
- room swipe navigation.

## 23. Reduced-motion contract

When reduced motion is active:

- lateral arcs become short direct interpolation;
- depth travel becomes restrained opacity/scale handoff plus compact reframe;
- cross-floor vertical movement becomes context replacement with minimal positional change;
- inspection remains directly manipulable;
- no idle camera drift is allowed;
- duration target is generally `180–320 ms` for route reframes;
- semantic hierarchy and focus timing remain unchanged.

## 24. Camera and overlays

Opening an overlay does not surrender camera ownership.

The camera may:

- remain static;
- apply a small registered background reframe;
- reduce parallax response;
- freeze inspection input.

It may not:

- wander decoratively;
- continue a nonessential orbit;
- move the selected object behind critical overlay text;
- change route context.

## 25. Asset-loading behavior

When final bounds are unavailable:

1. use registry metadata if available;
2. otherwise use profile/category proxy bounds;
3. frame the proxy immediately;
4. load and validate the final model;
5. resolve matched final bounds;
6. perform a compact proxy-to-final correction only when materially necessary.

Late model arrival must not cause a large unexpected camera jump.

## 26. Failure and recovery

### Invalid preset

- reject the preset;
- use the next fallback in resolution order;
- emit a diagnostic event.

### Invalid model bounds

- use category proxy bounds;
- disable inspection until validated;
- keep navigation operational.

### NaN or unstable pose

- immediately restore the last valid pose;
- cancel the active camera plan;
- emit a high-severity diagnostic;
- continue rendering the last valid frame.

### Renderer context recovery

- rebuild the renderer camera object from the authoritative Camera Engine state;
- do not reconstruct camera state from scene nodes.

## 27. Persistence

Persistable camera data:

- last semantic route target;
- optional last device hero orientation where enabled;
- user-authored preset overrides;
- editor-authored camera metadata.

Not persisted by default:

- transient inspection yaw/pitch;
- pinch zoom offsets;
- interrupted transition progress;
- temporary overlay reframe.

## 30. Performance requirements

The Camera Engine must:

- perform no model traversal during the render loop;
- use precomputed or cached bounds;
- avoid per-frame object allocation in interpolation paths;
- keep camera updates deterministic under Full, Balanced and Essential tiers;
- preserve direct-manipulation response at the current display refresh rate;
- reduce visual quality before reducing input responsiveness;
- recalculate safe framing only when relevant inputs change.

Exact timing and memory budgets are finalized in 3.5.6.

## 33. Acceptance criteria

3.5.2 is complete when:

1. One Camera Engine owns every runtime camera pose.
2. No feature component mutates the renderer camera directly.
3. Home, room, device, control, inspection, exploded and map modes are defined.
4. Every meaningful view resolves through a versioned preset and fallback chain.
5. Safe framing accounts for UI regions and iOS safe areas.
6. Responsive orientation changes reframe without replaying route transitions.
7. Route-level camera movement integrates with the Transition Director.
8. Direct manipulation arrives only as semantic intents from the Input Engine.
9. Inspection is constrained, resettable and conflict-free with capability gestures.
10. Missing assets or metadata never produce a blank or invalid frame.
11. Invalid poses recover to the last valid camera state.
12. Reduced-motion camera behavior preserves hierarchy with minimal travel.
13. Camera state can rebuild after renderer context loss.
14. Diagnostics expose pose, target, preset, safe frame and transition ownership.
15. Unit, integration and real-device validation requirements are explicit.

## 34. Engineering decisions

```text
CAM-001 The Camera Engine is the sole owner of mutable camera state
CAM-002 Route-level camera movement is coordinated by the Transition Director
CAM-003 UI components provide safe regions, never camera coordinates
CAM-004 Camera presets resolve through user, blueprint, exact-model and fallback layers
CAM-005 Zoom changes camera/projection, never device geometry
CAM-006 Raw pointer events never reach the Camera Engine
CAM-007 Camera roll is prohibited unless explicitly authored
CAM-008 Responsive changes reframe without replaying navigation transitions
CAM-009 Last valid pose is the mandatory recovery state
CAM-010 Missing model metadata falls back to validated proxy framing
```
