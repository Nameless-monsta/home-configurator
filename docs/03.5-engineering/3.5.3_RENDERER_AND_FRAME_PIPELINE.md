# 3.5.3 — Renderer and Frame Pipeline

**Status:** Approved engineering baseline  
**Phase:** 3.5 — Engineering Specifications  
**Depends on:** 3.5.1 Runtime Execution Model, 3.5.2 Camera Engine Contract, 3.9 Transition Library, 3.10 Animation Library

## 1. Purpose

Define the renderer ownership model, frame lifecycle, scene graph execution, post-processing policy, invalidation strategy, quality adaptation, failure handling and diagnostics for Home Configurator.

The renderer must preserve one persistent WebGL stage while rooms, devices, controls, scenes and overlays change around it.

## 2. Ownership boundary

```text
Application state
→ Presentation state
→ Scene graph
→ Renderer
→ Browser canvas
```

The Renderer owns:

- the WebGL renderer instance;
- the persistent canvas;
- scene roots and render layers;
- render targets and post-processing;
- frame scheduling;
- scene graph application;
- material and light updates;
- render quality application;
- resize and device-pixel-ratio handling;
- resource disposal;
- renderer diagnostics.

The Renderer does not own:

- Home Assistant state;
- semantic device capabilities;
- route state;
- camera intent;
- gesture arbitration;
- command reconciliation;
- transition selection;
- asset-registry matching;
- UI layout.

## 3. Renderer architecture

```text
RendererRuntime
├── CanvasHost
├── WebGLContextManager
├── SceneGraphAdapter

├── ResourceManager

```

## 4. Persistent canvas contract

The application mounts exactly one primary WebGL canvas for normal runtime.

Rules:

- route changes do not remount the canvas;
- room changes do not recreate the renderer;
- device changes replace scene nodes, not the canvas;
- overlays remain HTML/UI unless a true render-layer effect is required;
- WebGL context is reused until fatal loss or explicit disposal;
- context creation failure must fall back to a stable non-WebGL presentation.

## 5. Renderer lifecycle

```text
UNINITIALIZED
→ CREATING_CONTEXT
→ BUILDING_PIPELINE
→ READY
→ ACTIVE
→ SUSPENDED
→ RECOVERING
→ DISPOSING
→ DISPOSED
```

### READY

The renderer, canvas, camera binding and minimum scene are valid.

### ACTIVE

Frames are scheduled according to invalidation, transitions, gestures, animation and live state.

### SUSPENDED

Used when the document is hidden, the runtime is paused or the canvas is not visible.

### RECOVERING

Used after recoverable context loss or resource failure.

## 6. Public contract

```ts
interface RendererRuntime {
  initialize(options: RendererInitOptions): Promise<void>;
  attachSceneGraph(graph: SceneGraphSnapshot): void;
  applyScenePatch(patch: SceneGraphPatch): void;
  setCamera(camera: ResolvedCameraState): void;
  setQualityTier(tier: RenderQualityTier): void;
  invalidate(reason: RenderInvalidationReason): void;
  suspend(reason: RendererSuspendReason): void;
  resume(reason: RendererResumeReason): void;
  capture(options?: CaptureOptions): Promise<ImageBitmap | Blob>;
  getDiagnostics(): RendererDiagnosticsSnapshot;
  dispose(): Promise<void>;
}
```

## 7. Frame ownership

Only the Frame Scheduler may request animation frames.

Feature modules may call:

```text
invalidate(reason)
```

They may not start independent `requestAnimationFrame` loops.

This prevents:

- duplicate frame loops;
- unsynchronized animation clocks;
- hidden background work;
- frame-order races;
- inconsistent quality adaptation.

## 8. Frame scheduling modes

```text
IDLE_ON_DEMAND
CONTINUOUS_INTERACTION
CONTINUOUS_ANIMATION
CONTINUOUS_TRANSITION
THROTTLED_BACKGROUND
SUSPENDED
```

### IDLE_ON_DEMAND

Default stable mode. Render only when state, layout, lighting, camera, material or asset readiness changes.

### CONTINUOUS_INTERACTION

Used during direct manipulation, gesture preview, map movement or continuous controls.

### CONTINUOUS_ANIMATION

Used only while registered runtime animations are active.

### CONTINUOUS_TRANSITION

Used while a primary transition owns the viewport.

### THROTTLED_BACKGROUND

Optional low-rate updates for meaningful ambient effects when visible and allowed by quality and reduced-motion policy.

## 9. Canonical frame lifecycle

Every active frame follows this order:

```text
1. Read immutable runtime snapshot
2. Advance transition clock
3. Advance registered animation clocks
4. Resolve camera state
5. Resolve interaction-preview state
6. Apply scene graph patches
7. Apply model transforms
8. Apply material and emissive state
9. Apply lighting state
10. Update semantic effects and particles
11. Update post-processing uniforms
12. Perform culling and LOD selection
13. Render scene layers
14. Run post-processing passes
15. Publish frame diagnostics
16. Decide whether another frame is required
```

The order is fixed. A subsystem may skip a step but may not reorder the pipeline.

## 10. State snapshot rule

A frame consumes one immutable presentation snapshot.

State changes arriving during the frame are queued for the next frame.

This avoids:

- partial-state rendering;
- command reconciliation mid-frame;
- camera/model mismatches;
- non-deterministic tests.

## 11. Scene graph contract

```ts
interface SceneGraphSnapshot {
  version: number;
  rootIds: string[];
  nodes: Record<string, SceneNode>;
  materials: Record<string, MaterialState>;
  lights: Record<string, LightState>;
  effects: Record<string, EffectState>;
}
```

Each scene node has a stable ID.

```ts
interface SceneNode {
  id: string;
  type: "group" | "model" | "mesh" | "light" | "effect" | "anchor";
  parentId?: string;
  assetId?: string;
  transform: TransformState;
  visible: boolean;
  layer: RenderLayerId;
  lifecycle: "loading" | "ready" | "leaving" | "retained" | "disposed";
  metadata?: Record<string, unknown>;
}
```

The renderer never searches the scene by display name.

## 12. Scene graph patches

Patches are explicit and versioned.

```ts
interface SceneGraphPatch {
  baseVersion: number;
  nextVersion: number;
  add?: SceneNode[];
  update?: SceneNodeUpdate[];
  remove?: string[];
  materialUpdates?: MaterialStateUpdate[];
  lightUpdates?: LightStateUpdate[];
  effectUpdates?: EffectStateUpdate[];
}
```

If the base version does not match, the renderer requests a full snapshot rather than guessing patch order.

## 13. Render layers

Required baseline layers:

```text
BACKGROUND
ENVIRONMENT
ROOM
HERO_DEVICE
CONTROL_3D
SEMANTIC_EFFECTS
SPATIAL_LABELS
DEBUG
```

Rules:

- HTML interface chrome is not rendered in WebGL by default;
- control objects use `CONTROL_3D` only when genuine depth interaction is required;
- debug content is excluded from production captures and normal runtime;
- layer order is centrally registered.

## 14. Lighting pipeline

Lighting is semantic and authored.

Baseline components:

```text
Environment contribution
Key light
Fill light
Rim/accent light
Device-originated emissive light
Capability-specific semantic light
```

Rules:

- device-generated colour remains local;
- navigation and text chrome remain visually neutral;
- light counts are bounded by performance tier;
- physical lights may be represented by emissive materials without creating a dynamic light for every device;
- shadows are selective, not globally enabled for all objects;
- lighting state reconciles from confirmed or optimistic presentation state according to the runtime contract.

## 15. Material state pipeline

Material changes are applied through registered semantic channels:

```text
power
brightness
colour
availability
selection
pending
error
variant
environment
```

```ts
interface MaterialState {
  materialId: string;
  baseColor?: ColorValue;
  roughness?: number;
  metalness?: number;
  emissiveColor?: ColorValue;
  emissiveIntensity?: number;
  opacity?: number;
  transmission?: number;
  modifiers?: string[];
}
```

No feature module mutates Three.js material objects directly.

## 16. Post-processing pipeline

Allowed baseline passes:

```text
Tone mapping
Colour-space conversion
Selective bloom
Subtle vignette
 restrained grain/noise
Optional depth-based effects
```

Rules:

- post-processing must be modular and quality-tier aware;
- bloom is selective and tied to emissive sources;
- no full-screen decorative effects unrelated to state;
- no permanent green tint or network-line background;
- the background remains restrained and aligned with the approved monochrome tonal system;
- effects degrade before interaction responsiveness degrades.

## 17. Colour management

The renderer must use one declared colour-management path.

Requirements:

- texture colour space is declared explicitly;
- output colour space is consistent across devices;
- tone mapping is centralized;
- UI colour values and device-emission values are converted correctly;
- screenshots and diagnostics use the same output transform as the visible canvas;
- unsupported or ambiguous texture metadata produces a validation warning.

## 18. Device pixel ratio policy

DPR is controlled by the Quality Adapter.

```text
Full tier: bounded high DPR
Balanced tier: moderate DPR
Essential tier: conservative DPR
```

Rules:

- never use uncapped device DPR;
- adjust DPR gradually after sustained performance evidence;
- orientation and resize may trigger immediate safe recalculation;
- DPR changes must not recreate the full scene;
- touch coordinate mapping remains correct after DPR changes.

## 19. Resize and safe-area lifecycle

```text
viewport change
→ layout engine publishes safe frame
→ camera engine resolves new framing
→ renderer resizes drawing buffer
→ post-processing targets resize
→ one stabilization frame
→ input mapping confirms dimensions
```

Resize handling is debounced for layout work but must remain visually responsive.

## 20. Culling and LOD

The renderer applies:

- frustum culling;
- visibility culling;
- render-layer culling;
- model LOD selection;
- effect-distance culling;
- hidden-view suspension.

LOD selection considers:

```text
screen-space size
quality tier
device performance
transition state
interaction state
model importance
```

The hero device receives the highest available justified LOD.

## 21. Loading behavior

Loading hierarchy:

```text
last valid frame
→ proxy or category silhouette
→ low LOD or partial asset
→ final model
```

Rules:

- the canvas is never cleared to blank while a target asset loads;
- proxy and final model share pivot, scale and camera metadata;
- final replacement occurs only when required GPU resources are ready;
- loading indicators remain localized;
- adjacent likely targets may preload within budget.

## 22. Resource management

The Resource Manager tracks:

- geometries;
- textures;
- materials;
- render targets;
- shaders/programs;
- environment maps;
- model instances;
- effect buffers.

Each resource has:

```ts
interface ManagedResource {
  id: string;
  type: string;
  refCount: number;
  estimatedBytes: number;
  lastUsedFrame: number;
  disposable: boolean;
}
```

Rules:

- resources are reference counted;
- shared assets are not disposed with one device instance;
- transient targets are pooled where practical;
- inactive resources are evicted according to the performance budget;
- disposal occurs outside the critical visible frame where possible.

## 23. Shader policy

- shaders are registered assets;
- runtime string-generated shader code is prohibited in production plugins;
- shader variants are bounded;
- compilation is warmed where practical;
- a shader compile failure falls back to the nearest supported material;
- plugin shaders declare quality and capability requirements;
- custom shaders cannot bypass resource and frame diagnostics.

## 24. WebGL context loss

On context loss:

```text
prevent default browser disposal where supported
→ suspend frame scheduling
→ retain semantic/runtime state
→ show stable HTML fallback
→ rebuild renderer resources
→ restore scene graph
→ restore camera
→ resume with compact recovery transition
```

Context loss must not require Home Assistant rediscovery.

## 25. Visibility and background behavior

When the document becomes hidden:

- continuous frame loops stop;
- animation clocks record suspension;
- media progress remains source-state based;
- command and Home Assistant subscriptions continue according to runtime policy;
- on resume, confirmed state is reconciled before visual animation resumes.

## 26. Reduced-motion rendering

Reduced motion affects rendering as well as animation:

- particles stop or become static;
- temporal distortion effects disable;
- motion blur is prohibited;
- animated grain may become static;
- transition dissolves simplify;
- value and state changes remain immediately visible.

## 27. Quality tiers

```ts
type RenderQualityTier = "full" | "balanced" | "essential";
```

### Full

- authored shadows;
- selective bloom;
- approved particles;
- complete material response;
- highest justified LOD;
- smooth environment interpolation.

### Balanced

- reduced DPR;
- fewer shadow casters;
- reduced particles;
- simplified environment and material layers;
- medium LOD where not visually significant.

### Essential

- direct transforms and opacity;
- simple lighting;
- no expensive particles or procedural distortion;
- low-cost materials;
- responsive hero object and controls preserved.

State meaning must remain identical across tiers.

## 28. Error isolation

A failed node, material, texture or effect must remain local.

Examples:

- missing model → category proxy;
- failed texture → neutral material;
- failed shader → standard material;
- failed effect → effect disabled;
- failed supporting light → base environment retained;
- failed post-processing pass → render without that pass.

The renderer must not crash the full stage for one optional asset.

## 29. Accessibility implications

- the canvas has a semantic description supplied by the active experience;
- essential state also exists in accessible HTML;
- no rendered motion is the sole indication of state;
- focus remains in the HTML accessibility tree;
- renderer recovery does not reset keyboard focus;
- capture and developer overlays exclude private Home Assistant metadata by default.

## 30. Renderer events

```text
renderer:initializing
renderer:ready
```

Frame-level events are diagnostic-only and must not create high-frequency application-state updates.

## 31. Diagnostics

Required metrics:

- FPS and frame-time percentiles;
- CPU frame preparation time;
- GPU time where available;
- draw calls;
- triangles and visible objects;
- active lights and shadow casters;
- texture and geometry estimates;
- render-target memory;
- DPR and quality tier;
- active transition and animation counts;
- shader programs and compile failures;
- resource loads, cache hits and evictions;
- context loss/recovery count;
- long-frame reasons.

Developer overlays may show:

```text
FPS/frame graph
safe frame
camera frustum
scene node IDs
bounds and pivots
LOD level
render layers
draw-call attribution
light and shadow volumes
```

## 32. Testing contract

Required automated tests:

1. only one frame scheduler owns `requestAnimationFrame`;
2. frame steps execute in canonical order;
3. one immutable presentation snapshot is used per frame;
4. scene patches reject stale base versions;
5. route/device changes preserve the canvas and renderer instance;
6. material changes occur through semantic state channels;
7. resize preserves camera/input alignment;
8. quality-tier changes preserve state meaning;
9. hidden-view suspension stops continuous loops;
10. missing assets resolve to stable local fallbacks;
11. shared resources are not disposed prematurely;
12. context restoration rebuilds the same semantic scene;
13. reduced-motion mode disables nonessential effects;
14. optional post-processing failure does not block base rendering;
15. diagnostics identify long frames and resource ownership.

Required device tests:

- Safari on supported iPad hardware;
- iPhone portrait and landscape;
- desktop Chromium and Safari;
- orientation changes;
- rapid room/device switching;
- long-running idle session;
- context loss simulation;
- low-memory/resource eviction;
- full, balanced and essential quality tiers;
- reduced-motion mode;
- loading proxy to final-model replacement.

## 33. Acceptance criteria

3.5.3 is approved when:

1. one persistent renderer and canvas are defined;
2. only the Frame Scheduler owns frame requests;
3. the frame lifecycle is deterministic;
4. scene graph snapshots and patches are versioned;
5. render layers and ownership are explicit;
6. lighting, materials and post-processing are semantic and bounded;
7. DPR, resize, culling and LOD behavior are defined;
8. loading never produces a blank stage;
9. resources have measurable ownership and disposal rules;
10. WebGL context loss has a recovery path;
11. reduced-motion and quality tiers preserve state meaning;
12. failures remain local to optional nodes or effects;
13. diagnostics can explain frame cost and resource usage;
14. implementation can proceed without inventing renderer ownership or frame ordering.
