# 3.5.4 — Input and Gesture Arbitration

**Status:** Approved engineering baseline  
**Phase:** 3.5 — Engineering Specifications  
**Depends on:** 3.5.1 Runtime Execution Model, 3.5.2 Camera Engine Contract, 3.5.3 Renderer and Frame Pipeline, Phase 3 UX specifications

## 1. Purpose

The Input Engine owns all pointer, touch, pen, wheel, keyboard and assistive-interaction routing in Home Configurator.

No control, plugin, scene object, device module or renderer component may attach independent global gesture logic or interpret raw multi-touch input outside this engine.

The objective is to guarantee:

- deterministic gesture ownership;
- conflict-free touch interaction on iPhone and iPad;
- one consistent mapping from physical input to semantic intent;
- accessible alternatives for every gesture;
- interruption and cancellation rules;
- stable behavior during transitions, overlays and pending commands;
- diagnostics for ambiguous or rejected input.

## 2. Ownership boundary

```text
Browser input event
→ Input Engine
→ candidate recognizers
→ arbitration
→ gesture session
→ semantic intent
→ runtime/controller
```

The Input Engine owns:

- event normalization;
- pointer tracking;
- gesture recognition;
- gesture arbitration;
- capture and release;
- cancellation;
- focus-aware keyboard routing;
- wheel routing;
- touch-action policy;
- hit-test coordination;
- accessibility fallback routing;
- input diagnostics.

It does not own:

- route changes;
- Home Assistant service calls;
- camera pose mutation;
- object animation;
- device capability detection;
- control values;
- UI layout.

## 3. Normalized input model

```ts
interface NormalizedPointer {
  id: number;
  type: "mouse" | "touch" | "pen";
  phase: "down" | "move" | "up" | "cancel";
  x: number;
  y: number;
  pressure: number;
  buttons: number;
  timestamp: number;
}
```

All browser-specific events are converted before recognition.

## 4. Input layers

Priority order:

```text
modal confirmation
> navigation overlay
> focused control
> map/editor surface
> selected 3D object
> room/background gesture
> ambient stage
```

A lower layer cannot claim input already captured by a higher layer.

## 5. Gesture session lifecycle

```text
IDLE
→ CANDIDATE
→ CLAIMED
→ ACTIVE
→ COMMITTING
→ COMPLETE
```

Alternative termination:

```text
CANDIDATE/CLAIMED/ACTIVE
→ CANCELED
```

A session records:

```ts
interface GestureSession {
  id: string;
  ownerId: string;
  recognizerId: string;
  pointers: number[];
  startedAt: number;
  state: "candidate" | "claimed" | "active" | "committing" | "complete" | "canceled";
  targetId?: string;
  capability?: string;
}
```


Plugins may register semantic recognizers only through the extension API. They may not install raw global listeners.

## 7. Arbitration rules

1. Explicit active-control mode wins over generic object manipulation.
2. Modal and overlay surfaces capture before the WebGL stage.
3. A recognizer must meet threshold before claiming input.
4. Pointer-count changes trigger safe re-arbitration or cancellation.
5. Once committed, a gesture cannot silently change semantic meaning.
6. Gesture ownership is exclusive unless a documented cooperative recognizer pair is used.
7. Route transitions may suspend new gesture claims while allowing safe release of the current session.
8. Repeated taps are debounced at the semantic-action layer, not by dropping raw events blindly.

## 8. Core mappings

### Device inspection

```text
one-finger drag on selected object → orbit/manipulate object
pinch in inspection mode → bounded zoom
```

### Color sphere

```text
one-finger drag → hue/saturation rotation
two-finger vertical motion → brightness
```

Sphere geometry never changes size to represent brightness.

### Climate

```text
one-finger drag on active temperature control → target adjustment
```

### Media

```text
vertical gesture in volume mode → volume
horizontal gesture in seek mode → seek
```

### Vacuum map

```text
one-finger drag on map → map or zone interaction depending on mode
two-finger gesture → viewport pan/zoom
```

### Room navigation

Edge swipe is optional and disabled by default in v1 unless testing proves it conflict-free.

## 9. Hit testing

Hit-test order:

```text
HTML/control target
→ semantic overlay target
→ 3D interactive node
→ stage background
```

The renderer returns semantic node IDs, never raw mesh references to feature code.

## 10. Gesture thresholds

Thresholds are design tokens and device-density aware.

Baseline guidance:

```text
tap travel tolerance: 6–10 CSS px
long press: 450–650 ms
drag claim: 6–12 CSS px
edge swipe origin: within 20–28 CSS px of edge
```

Final thresholds require iOS device testing.

## 11. Pointer capture

- claimed gestures use pointer capture where supported;
- capture is released on completion, cancellation, context loss or target disposal;
- hidden or unmounted targets force cancellation;
- pointer capture must not trap browser/system gestures outside the intended control region.

## 12. Scroll and browser gesture policy

The full-screen runtime prevents page scroll only inside the application surface.

Rules:

- `touch-action` is assigned per interaction region;
- browser navigation gestures are not disabled globally without need;
- iOS safe-area and system-edge gestures are respected;
- wheel input is active only when the focused control supports it;
- accidental page zoom is prevented only inside explicit full-screen interaction mode.

## 13. Keyboard routing

Focus order remains DOM-driven.

Suggested semantic actions:

```text
Enter/Space → activate
Escape → cancel/close/back
Arrow keys → adjust or navigate according to focused control
Page Up/Down → larger value step or floor navigation where defined
Home/End → bounds
```

Global shortcuts never override focused text inputs or browser-critical commands.

## 14. Accessibility contract

Every gesture must expose an equivalent semantic control.

Examples:

- color sphere → hue, saturation and brightness sliders;
- object orbit → rotate-left/right/up/down controls or preset views;
- map region selection → semantic list;
- temperature drag → slider and plus/minus controls;
- volume gesture → slider and mute button.

Screen-reader users interact with DOM controls, not gesture-only canvas regions.

## 15. Transition interaction policy

During a primary route transition:

- current active gesture is allowed to finish only when safe;
- new stage gestures are blocked during the short commit section;
- navigation cancel/back remains available where state integrity allows;
- queued input collapses to the latest valid semantic intent;
- focus is restored after the target is interactive.

## 16. Pending-command behavior

A pending command does not automatically lock all input.

Rules:

- related continuous gestures may coalesce;
- destructive duplicate actions are blocked;
- unrelated capabilities remain interactive;
- authoritative failure cancels only the affected optimistic interaction.

## 17. Error and cancellation behavior

Cancellation reasons include:

```text
pointer cancel
owner disposed
route commit
modal takeover
pointer-count mismatch
context loss
invalid capability
system interruption
```

Cancellation must:

- stop preview updates;
- release capture;
- restore confirmed state or the last safe preview;
- emit a diagnostic event;
- avoid sending a final command unless explicitly permitted.

## 20. Diagnostics

Developer mode exposes:

- active pointers;
- recognizer candidates;
- current owner;
- thresholds crossed;
- claim and rejection reasons;
- gesture velocity and distance;
- pointer capture state;
- semantic intent output;
- canceled-session history.

## 21. Performance requirements

- pointer normalization and arbitration must stay below the frame budget;
- continuous gesture preview must remain frame-rate independent;
- raw move events may be coalesced per animation frame;
- final release events are never dropped;
- gesture processing must not trigger layout thrashing.

## 22. Required tests

1. One-finger sphere rotation never activates brightness.
2. Two-finger brightness never rotates or scales sphere geometry.
3. Climate adjustment never rotates the hero object.
4. Map gestures do not trigger room navigation.
5. Overlay interaction always wins over stage interaction.
6. Pointer-count changes cancel or re-arbitrate safely.
7. Transition commits cannot leave captured pointers behind.
8. Keyboard and accessible alternatives perform the same semantic action.
9. iOS edge gestures remain usable outside explicit app controls.
10. Rapid repeated input collapses without duplicate unsafe commands.

## 23. Acceptance criteria

3.5.4 is approved when:

- all runtime input passes through one engine;
- gesture ownership is deterministic;
- each approved gesture has a semantic fallback;
- multi-touch conflicts are explicitly resolved;
- cancellation never leaves stale capture or optimistic state;
- overlays, controls, maps and 3D objects follow one priority model;
- tests cover iPad and iPhone touch behavior;
- diagnostics can explain why any gesture was accepted or rejected.
