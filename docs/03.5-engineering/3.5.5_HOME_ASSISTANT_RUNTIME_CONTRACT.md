# 3.5.5 — Home Assistant Runtime Contract

**Status:** Approved engineering baseline  
**Phase:** 3.5 — Engineering Specifications  
**Depends on:** 3.5.1 Runtime Execution Model, 2.3 Backend/Integration Architecture, 2.4 Data Model, 2.5 Entity Abstraction, 2.6 Capability Mapping

## 1. Purpose

The Home Assistant Runtime Adapter is the only subsystem allowed to communicate directly with Home Assistant APIs.

Its responsibilities are to:

- establish and maintain the Home Assistant connection;
- authenticate through supported Home Assistant mechanisms;
- fetch registries and initial state;
- subscribe to live state updates;
- normalize Home Assistant data into canonical runtime models;
- dispatch semantic commands as Home Assistant service calls;
- reconcile optimistic state with authoritative state;
- handle disconnect, reconnect, stale data, retries and failures;
- expose diagnostics without leaking secrets.

No renderer, control, plugin or device experience may call Home Assistant services directly.

## 2. Ownership boundary

```text
Home Assistant WebSocket / REST
→ HA Runtime Adapter
→ normalization and discovery
→ confirmed runtime state
→ capability layer
→ UI and renderer
```

Commands flow in the opposite direction:

```text
semantic intent
→ command queue
→ HA Runtime Adapter
→ service call
→ event/state confirmation
→ reconciliation
```

The adapter owns:

- connection lifecycle;
- authentication session use;
- registry/state ingestion;
- subscriptions;
- service-call translation;
- HA-specific feature flags and service metadata;
- correlation and confirmation evidence;
- reconnect and resubscription;
- stale-state markers;
- adapter diagnostics.

It does not own:

- UX state;
- optimistic presentation rules;
- gesture handling;
- renderer state;
- device ordering;
- blueprint presentation;
- model selection.

## 3. Connection states

```text
UNINITIALIZED
CONNECTING
AUTHENTICATING
SYNCING
READY
DEGRADED
RECONNECTING
DISCONNECTED
AUTH_FAILED
FATAL
```

### READY

- initial registries and entity state are loaded;
- required subscriptions are active;
- commands may be dispatched.

### DEGRADED

- the connection exists but one or more noncritical registries, subscriptions or integrations failed;
- unaffected devices remain usable.

### RECONNECTING

- confirmed state is retained but marked stale according to policy;
- new commands are rejected, queued or conditionally accepted by command policy;
- exponential backoff is active.

## 4. Bootstrap sequence

```text
resolve HA environment
→ open connection
→ authenticate
→ fetch config and capabilities
→ fetch area/device/entity registries
→ fetch current states
→ build canonical snapshot
→ subscribe to state and registry changes
→ publish READY
```

The runtime may present a proxy or cached experience before READY, but it must not label cached values as live.

## 5. Environment detection

Supported deployment contexts include:

- Home Assistant frontend/custom panel;
- HACS-installed frontend resource;
- authenticated same-origin runtime;
- development runtime using an explicitly configured HA URL and token.

Production UI must never ask users to paste a long-lived access token when a native Home Assistant authentication context is available.

## 6. Initial synchronization

The adapter gathers, when available:

- Home Assistant configuration;
- areas and floors;
- devices;
- entities;
- entity states;
- services;
- labels/categories used by discovery;
- user-visible names and disabled/hidden metadata.

Initial data is converted into one immutable confirmed-state snapshot before publication.

## 7. Subscription model

Required live inputs:

```text
state_changed
entity_registry updates
device_registry updates
area_registry updates
floor_registry updates where available
connection status
```

Optional integration-specific events may be registered through adapter plugins.

Subscriptions are centrally tracked and automatically restored after reconnect.

## 8. Canonical state update flow

```text
HA event
→ validate payload
→ update HA cache
→ normalize affected entity/device
→ recompute affected capabilities only
→ publish confirmed-state patch
→ command reconciliation
→ renderer/UI consume next snapshot
```

A single entity event must not force complete rediscovery of the installation.

## 9. State validity and staleness

Each confirmed value carries evidence:

```ts
interface ConfirmedValue<T> {
  value: T;
  observedAt: number;
  sourceEntityId?: string;
  available: boolean;
  stale: boolean;
}
```

Rules:

- `unavailable` and `unknown` are explicit states, not missing values;
- last-known values may remain visible only with stale labeling;
- a reconnect does not silently turn stale data into confirmed live data;
- state becomes fresh only after successful sync or event confirmation.

## 10. Semantic command contract

The runtime sends semantic commands, for example:

```ts
interface SemanticCommand {
  id: string;
  deviceId: string;
  capability: string;
  action: string;
  value?: unknown;
  issuedAt: number;
  policy: CommandPolicy;
}
```

The HA adapter resolves this into one or more service calls using the resolved device profile and entity bindings.

Example:

```text
SetBrightness(device, 0.62)
→ light.turn_on
→ entity_id + brightness percentage/native value
```

## 11. Service-call translation

Translation priority:

```text
exact integration adapter
→ manufacturer/device plugin
→ capability-domain adapter
→ generic Home Assistant adapter
→ unsupported
```

The adapter must validate:

- target entity still exists;
- capability remains supported;
- value is within normalized limits;
- required service fields exist;
- command is permitted in current availability state.

## 12. Command dispatch states

```text
QUEUED
DISPATCHING
ACKNOWLEDGED
AWAITING_CONFIRMATION
CONFIRMED
PARTIALLY_CONFIRMED
FAILED
TIMED_OUT
CANCELED
```

A successful service-call response is acknowledgement, not necessarily final confirmation.

## 13. Confirmation strategy

Confirmation evidence, strongest first:

```text
1. matching authoritative state update
2. integration-specific completion event
3. matching service response with verified synchronous semantics
4. timeout followed by explicit state refresh
```

The runtime must not claim confirmation solely because a service request returned HTTP/WebSocket success when the expected state is asynchronous.

## 14. Optimistic reconciliation

```text
optimistic preview
→ command dispatch
→ acknowledgement
→ authoritative event
→ compare expected and actual
→ confirm or reconcile
```

Rules:

- matching state settles smoothly;
- differing state animates to the authoritative value;
- unrelated state changes never cancel the command;
- superseded continuous commands reconcile against the newest command only;
- failed sub-capabilities do not invalidate the whole composed device.

## 15. Continuous-command coalescing

Used for brightness, color, volume and temperature.

- preview updates may occur every frame;
- outbound service calls are throttled by capability policy;
- queued intermediate values may be replaced by the latest value;
- final release value is dispatched immediately;
- every dispatch retains a traceable command ID.

## 16. Retry policy

Automatic retry is allowed only when:

- the action is idempotent or explicitly retry-safe;
- failure is transport-related;
- the command has not been superseded;
- retry remains inside the configured timeout window.

No automatic retry for:

- high-risk security actions;
- non-idempotent scripts;
- commands rejected by Home Assistant validation;
- actions with unknown external side effects.

## 17. Reconnect policy

Backoff uses bounded exponential delay with jitter.

```text
connection lost
→ mark adapter RECONNECTING
→ retain last confirmed snapshot as stale
→ cancel or hold commands by policy
→ reconnect
→ authenticate
→ refresh registries and states
→ restore subscriptions
→ reconcile retained runtime state
→ READY
```

A reconnect must not duplicate subscriptions or replay unsafe commands.

## 18. Offline and disconnected behavior

Navigation, model inspection and cached presentation may remain operational.

Controls behave according to policy:

```text
read-only
queue-safe commands
reject commands
```

Default v1 recommendation:

- reject device commands while disconnected;
- preserve local navigation and inspection;
- clearly show stale values;
- do not silently queue actions for later execution unless the user explicitly enables it.

## 19. Registry changes

When areas, devices or entities change:

- stable internal IDs are preserved where mapping evidence remains valid;
- affected discovery results are recalculated incrementally;
- user overrides remain authoritative;
- new devices enter review or auto-admission according to discovery policy;
- removed entities invalidate only their bindings;
- navigation is not reordered during active use without an explicit safe update point.

## 20. Partial device availability

A composed device may contain several entities.

Example:

```text
TV entity available
Apple TV entity unavailable
receiver available
```

The runtime must:

- preserve available controls;
- disable only capabilities owned by unavailable bindings;
- report the affected source;
- avoid collapsing the entire device into unavailable state unless no meaningful capability remains.

## 21. Error model

```ts
interface HAAdapterError {
  code: string;
  category:
    | "authentication"
    | "connection"
    | "registry"
    | "subscription"
    | "service"
    | "validation"
    | "timeout"
    | "integration";
  recoverable: boolean;
  userMessage: string;
  technicalMessage?: string;
  commandId?: string;
  entityId?: string;
}
```

Secrets and tokens are never included in logs or user-facing diagnostics.

## 22. Caching

Allowed local cache:

- normalized registry metadata;
- model and blueprint mappings;
- last confirmed entity state for startup continuity;
- adapter capability metadata.

Rules:

- cache is versioned;
- stale state is marked;
- authentication credentials are not stored by the frontend outside supported HA mechanisms;
- corrupted cache is discarded safely;
- cache is optional and resettable.

## 23. Security and privacy

- all communication remains local unless the user’s HA endpoint itself is remote;
- no telemetry by default;
- entity IDs, room names, maps and state data are never sent to external services;
- diagnostic exports redact tokens, URLs where sensitive, map coordinates and personal labels by default;
- plugin adapters receive only declared capability and binding access.

## 24. Public interface

```ts
interface HomeAssistantRuntime {
  connect(): Promise<void>;
  disconnect(): Promise<void>;
  getStatus(): HAConnectionStatus;
  getConfirmedSnapshot(): ConfirmedRuntimeSnapshot;
  subscribe(listener: (patch: ConfirmedStatePatch) => void): Unsubscribe;
  dispatch(command: SemanticCommand): Promise<CommandReceipt>;
  refresh(scope?: RefreshScope): Promise<void>;
}
```

## 26. Diagnostics

Developer mode exposes:

- connection state and latency;
- last successful sync;
- subscription count;
- event throughput;
- stale entity count;
- registry versions;
- pending commands;
- command translation result;
- confirmation evidence;
- retry and timeout history;
- reconnect attempts.

Sensitive values remain redacted.

## 27. Required tests

1. Initial sync publishes one coherent confirmed snapshot.
2. State events update only affected devices/capabilities.
3. Service acknowledgement is not treated as confirmation when state is asynchronous.
4. Continuous commands coalesce and final release is preserved.
5. Disconnect marks values stale and prevents unsafe dispatch.
6. Reconnect restores subscriptions without duplicates.
7. Failed supporting entities do not disable a composed device.
8. Registry removal invalidates only affected bindings.
9. Authentication failure never exposes credentials.
10. Out-of-order confirmations reconcile against the newest valid command.
11. Unknown/unavailable states remain explicit.
12. Cached startup never presents stale values as live.

## 28. Acceptance criteria

3.5.5 is approved when:

- all HA communication passes through one runtime adapter;
- initial sync and subscriptions produce deterministic confirmed state;
- semantic commands are translated without UI knowledge of HA domains;
- optimistic values reconcile against authoritative evidence;
- disconnect, reconnect, retry, stale-state and timeout behavior are explicit;
- composed devices tolerate partial entity failure;
- registry changes preserve user overrides and stable mappings;
- credentials and private home data remain protected;
- diagnostics can trace every command from semantic intent to confirmation or failure.
