# 3.5.6 — Performance and Resource Budgets

**Status:** Approved engineering baseline  
**Phase:** 3.5 — Engineering Specifications  
**Depends on:** 3.5.1 Runtime Execution Model, 3.5.2 Camera Engine Contract, 3.5.3 Renderer and Frame Pipeline, 3.5.4 Input and Gesture Arbitration

## 1. Purpose

This contract defines measurable runtime budgets for frame time, responsiveness, memory, GPU resources, assets, network activity and background work.

Performance degradation must reduce visual quality before it reduces input responsiveness, command reliability, accessibility or state accuracy.

## 2. Target devices

The canonical validation target is a recent iPad running Safari in landscape orientation.

Supported classes:

```text
A — iPad / modern desktop
B — recent iPhone / balanced laptop
C — older mobile / constrained embedded browser
```

Device class is determined at runtime from measured behavior, not browser-name assumptions alone.

## 3. Frame-rate targets

```text
Class A target: 60 FPS
Class B target: 45–60 FPS
Class C target: stable 30 FPS minimum
```

Frame-time guidance:

```text
60 FPS budget: 16.67 ms
45 FPS budget: 22.22 ms
30 FPS budget: 33.33 ms
```

The application must prefer stable cadence over oscillating quality.

## 4. Frame budget allocation

Nominal Class A frame:

```text
Input and gesture processing       ≤ 1.0 ms
State snapshot and reconciliation  ≤ 1.5 ms
Animation and transition update    ≤ 2.0 ms
Scene graph update                 ≤ 2.0 ms
Culling and LOD                    ≤ 1.5 ms
Render submission                  ≤ 6.5 ms
Post-processing                    ≤ 1.5 ms
Diagnostics and margin             remainder
```

Budgets are directional, not per-frame hard failures. Sustained overruns trigger quality adaptation.

## 5. Input latency

Required targets:

```text
Pointer/touch visual acknowledgement: ≤ 50 ms
Direct manipulation response: same rendered frame where possible
Control value preview: ≤ 50 ms
Navigation trigger acknowledgement: ≤ 80 ms
```

No asset load, analytics task, shader compilation or background discovery may block active input.

## 6. Quality adaptation

Quality tiers:

```text
FULL
BALANCED
ESSENTIAL
```

Adaptation order:

1. reduce particle count and update rate;
2. reduce shadow quality;
3. reduce bloom and post-processing;
4. reduce environment-map resolution;
5. lower dynamic DPR;
6. simplify material features;
7. select lower LOD;
8. suspend nonessential ambient animation.

Never degrade:

- semantic state;
- control availability;
- text clarity;
- focus visibility;
- touch target size;
- command dispatch or reconciliation.

## 7. Dynamic resolution

Device pixel ratio is capped by quality tier and measured frame stability.

Recommended starting caps:

```text
FULL:      min(device DPR, 2.0)
BALANCED:  min(device DPR, 1.5)
ESSENTIAL: 1.0
```

DPR changes:

- occur gradually;
- are suppressed during precision gestures;
- do not replay route transitions;
- preserve CSS layout and hit testing;
- are recorded in diagnostics.

## 8. Stability thresholds

A quality downgrade may occur when either condition persists:

```text
Average frame time exceeds tier budget for 2 seconds
or
More than 10% of frames exceed 1.5× target over 3 seconds
```

Upgrade requires a longer stable window:

```text
At least 10 seconds below 75% of the active tier budget
```

Hysteresis prevents rapid tier oscillation.

## 9. JavaScript main-thread budget

Long tasks above 50 ms are unacceptable during active interaction.

Rules:

- asset parsing uses workers where practical;
- discovery normalization is chunked;
- large schema validation is incremental;
- model decompression does not run synchronously during gestures;
- idle work uses explicit scheduling and cancellation;
- hidden tabs suspend nonessential loops.

## 10. Memory budgets

Guidance for active runtime:

```text
Class A working set target: ≤ 350 MB
Class B working set target: ≤ 250 MB
Class C working set target: ≤ 160 MB
```

These are browser-runtime targets, not guaranteed OS limits.

The Resource Manager tracks:

- decoded textures;
- geometry buffers;
- render targets;
- material and shader variants;
- cached models;
- maps and media artwork;
- retained state snapshots;
- worker memory.

## 11. GPU resource budgets

Class A active scene guidance:

```text
Draw calls:             target ≤ 120, hard review at 180
Visible triangles:      target ≤ 750k, hard review at 1.2M
Material programs:      target ≤ 32 active variants
Shadow-casting lights:  target ≤ 2
Full-screen passes:     target ≤ 4
```

Class B and C budgets scale down through the active quality tier.

The renderer must expose actual counts in developer diagnostics.

## 12. Texture budgets

Default maximum texture dimensions:

```text
Hero object critical texture: 2048 px
Supporting object texture:     1024 px
Thumbnail / UI asset:           512 px
Environment map:               tier-dependent
```

Rules:

- use compressed GPU textures where supported;
- avoid uncompressed 4K assets in normal runtime;
- combine channels where appropriate;
- generate mipmaps unless the usage explicitly forbids them;
- texture memory is estimated before activation;
- oversized community assets are rejected or downscaled by the asset pipeline.

## 13. Model residency and cache

Resident set should normally include:

```text
Current hero model
Current environment/room
Previous and next likely model
Shared generic fallbacks
Active control assets
```

Rules:

- preloading is prediction-based and cancellable;
- exact-device models never displace required fallbacks;
- least-recently-used assets are evicted first;
- active transition assets remain pinned until settle;
- disposed GPU resources must not remain referenced by caches.

## 14. Asset load budgets

Compressed transfer guidance:

```text
Initial shell and runtime JS:        target ≤ 500 KB gzip, excluding HA host assets
Initial visual bootstrap assets:     target ≤ 1.5 MB
Typical generic device model:        target ≤ 1.5 MB
Detailed hero model:                 target ≤ 4 MB
Optional model pack assets:          lazy loaded
```

Large models may exceed the guidance only with lower-LOD fallbacks and explicit validation.

## 15. Startup budgets

On a warm local Home Assistant connection:

```text
Shell visible:                    ≤ 700 ms
Stable proxy scene:              ≤ 1.2 s
First interactive navigation:    ≤ 1.5 s
Final hero asset:                progressive; must not block interaction
```

The application must render a valid proxy state before final assets are ready.

## 16. Network and Home Assistant budgets

- entity subscriptions are consolidated;
- service calls are throttled and coalesced per capability;
- no polling is introduced when subscription data is available;
- model packs and static assets are cached locally;
- runtime remains usable with the last confirmed state during reconnect;
- media artwork and maps have bounded cache policies.

Network latency does not block local gesture preview.

## 17. Continuous-control budgets

During brightness, color, volume or temperature manipulation:

```text
Visual preview: every rendered frame
Semantic value update: every frame or coalesced snapshot
HA command transmission: capability-specific throttle
Final command: immediate on release
```

The command throttle must never be implemented by delaying visual response.

## 18. Transition budgets

- only one primary viewport transition runs at once;
- expensive post-processing may reduce during transitions;
- shader compilation must be warmed before a transition where practical;
- route commit sections remain below 160 ms;
- frame drops must not leave the scene between valid states.

## 19. Background and hidden-state behavior

When the document is hidden:

- rendering stops or drops to a minimal maintenance cadence;
- ambient loops pause;
- Home Assistant subscriptions remain active where supported;
- confirmed state continues updating;
- no animation backlog is replayed on return;
- current view reconciles directly to the latest state.

## 20. Thermal and battery response

On sustained thermal pressure or battery-saving mode:

- quality may move to BALANCED or ESSENTIAL;
- target FPS may become 30;
- post-processing and particles reduce first;
- semantic device animation remains where inexpensive;
- critical controls and state feedback remain unchanged.

## 21. Accessibility performance

- reduced-motion mode removes nonessential animation cost;
- VoiceOver focus updates must not wait for final 3D asset loading;
- keyboard input must remain responsive during scene changes;
- accessible HTML alternatives are available even if WebGL enters recovery mode.

## 22. Diagnostics contract

```ts
interface PerformanceSnapshot {
  timestamp: number;
  qualityTier: "full" | "balanced" | "essential";
  fps: number;
  frameTimeAverageMs: number;
  frameTimeP95Ms: number;
  longTasks: number;
  drawCalls: number;
  triangles: number;
  textures: number;
  geometries: number;
  gpuMemoryEstimateBytes?: number;
  jsHeapBytes?: number;
  dpr: number;
  activeModels: string[];
  downgradeReason?: string;
}
```

Developer mode exposes live and rolling values without materially affecting production performance.

## 23. Testing requirements

Required tests include:

1. iPad landscape sustained interaction test;
2. 10-minute memory and resource-leak test;
3. rapid room/device switching test;
4. repeated color-sphere manipulation test;
5. WebGL context-loss and recovery test;
6. hidden-tab resume test;
7. low-memory asset eviction test;
8. FULL-to-ESSENTIAL degradation test;
9. reduced-motion and VoiceOver performance test;
10. command latency under renderer load.

## 24. Acceptance criteria

3.5.6 is approved when:

1. frame-rate and frame-time targets are explicit;
2. input latency remains protected above visual quality;
3. quality tiers and downgrade order are deterministic;
4. DPR adaptation has hysteresis and gesture safeguards;
5. CPU, memory, GPU, texture and asset budgets are measurable;
6. startup and proxy-render targets are defined;
7. hidden-tab, thermal and battery behavior are defined;
8. continuous controls separate visual frequency from HA command frequency;
9. diagnostics expose all load-bearing measurements;
10. implementation tests can objectively pass or fail.
