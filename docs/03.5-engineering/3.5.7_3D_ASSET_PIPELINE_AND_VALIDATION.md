# 3.5.7 — 3D Asset Pipeline and Validation

**Status:** Approved engineering baseline  
**Phase:** 3.5 — Engineering Specifications  
**Depends on:** 2.7 Model Registry, 3.5.2 Camera Engine Contract, 3.5.3 Renderer and Frame Pipeline, 3.5.6 Performance and Resource Budgets

## 1. Purpose

This contract defines how 3D assets enter Home Configurator, how they are normalized, optimized, validated, versioned, registered, loaded and safely rejected.

A model is not runtime-ready merely because it opens in a GLTF viewer. Every asset must satisfy geometry, scale, pivot, material, animation, metadata, licensing and performance requirements.

## 2. Canonical runtime format

The canonical runtime format is binary glTF:

```text
.glb
```

Source formats may include:

```text
.blend
.fbx
.obj
.stl
.gltf
.cad-derived exports
```

Source formats are never loaded directly by the production runtime. They must pass through the asset pipeline first.

## 3. Pipeline stages

```text
Acquire
→ Inspect
→ Normalize
→ Repair
→ Optimize
→ Author metadata
→ Generate LODs
→ Compress
→ Validate
→ Package
→ Register
→ Publish
```

Every stage produces structured diagnostics.

## 4. Ownership

The Asset Pipeline owns:

- format conversion;
- coordinate and unit normalization;
- pivot and bounds validation;
- mesh cleanup;
- material normalization;
- texture processing;
- animation validation;
- LOD generation;
- compression;
- metadata generation;
- manifest validation;
- package integrity;
- licensing records.

The runtime owns:

- model resolution through the registry;
- loading approved runtime files;
- applying state mappings;
- selecting LODs;
- disposing resources.

The runtime must not repair malformed models dynamically beyond safe fallback behavior.

## 5. Units and coordinate system

Canonical rules:

```text
Units: metres
Up axis: +Y
Forward reference: -Z
Right-handed coordinates
```

Every model must have a physically plausible real-world scale.

Examples:

```text
Light bulb: approximately 0.05–0.15 m
Robot vacuum: approximately 0.30–0.40 m diameter
Television: derived from declared screen size
Wall switch: approximately 0.07–0.12 m
```

Implausible dimensions fail validation unless an explicit abstract-model flag is set.

## 6. Origin and pivot policy

Default origin:

```text
X/Z: horizontal center of stable footprint
Y: lowest intended contact plane
```

Exceptions require metadata.

Required pivot classes:

- world/root pivot;
- inspection pivot;
- animated-part pivots;
- optional camera/look-at anchors;
- optional emission/source anchors.

A vacuum model, for example, should rotate around its physical center while resting on the floor plane.

## 7. Scene hierarchy

Required root structure:

```text
HC_MODEL_ROOT
├── VISUAL
├── COLLISION          optional
├── ANCHORS            optional
├── ANIMATED_PARTS     optional
└── HELPERS            stripped from runtime unless declared
```

Rules:

- node names are stable and unique;
- model-pack updates preserve public node IDs where possible;
- hidden editor helpers are removed;
- transforms are applied or intentionally documented;
- negative scale is prohibited in final runtime nodes;
- nested depth should remain reasonable.

## 8. Naming convention

Node names use lower snake case or registered semantic IDs.

Examples:

```text
body
screen
bulb_emitter
status_led
main_brush
side_brush_left
lidar
power_button
vent_left
```

Public nodes referenced by capability mappings must be declared in the manifest.

## 9. Geometry rules

Required:

- valid indexed triangles;
- no NaN or infinite values;
- no zero-area triangles above tolerance;
- no accidental duplicate shells;
- consistent winding;
- valid normals;
- tangents generated only when required;
- hidden/internal geometry removed unless functionally necessary;
- mesh density proportional to visible importance.

Non-manifold geometry is permitted only where harmless to rendering and explicitly accepted by validation.

## 10. Polygon budgets

Guidance per LOD:

```text
LOD0 hero:       80k–250k triangles typical
LOD1 medium:     30k–100k
LOD2 low:         8k–35k
LOD3 proxy:       500–8k or authored silhouette
```

Higher counts require explicit exception metadata and must remain inside the active-scene performance budget.

A small switch or lamp must not consume the same geometry budget as a detailed vacuum dock system.

## 11. LOD contract

Each detailed model should provide:

```text
LOD0 — close hero/inspection
LOD1 — normal hero
LOD2 — room/supporting view
LOD3 — proxy or distant representation
```

Rules:

- LODs preserve silhouette and material identity;
- pivots and bounds remain aligned;
- public anchors and semantic nodes remain resolvable;
- switching must not visibly jump position or scale;
- animated models may use simplified authored animations on lower LODs;
- missing LODs trigger pipeline warnings or failure according to model class.

## 12. Materials

Preferred material model:

```text
Metallic-roughness PBR
```

Allowed features:

- base color;
- normal;
- metallic/roughness;
- occlusion;
- emissive;
- transmission for justified glass;
- clearcoat where materially important.

Rules:

- no baked green accent or theme-specific UI color;
- emissive surfaces correspond to plausible device sources;
- material count is minimized;
- shared materials are reused;
- shader-heavy custom extensions require explicit runtime support;
- unsupported extensions fail or degrade through a declared fallback.

## 13. Texture processing

Pipeline operations:

- resize to approved maximum dimensions;
- convert to supported compressed texture formats where available;
- generate mipmaps;
- pack compatible channels;
- remove unused alpha;
- validate color space;
- detect duplicate textures;
- calculate decoded GPU-memory cost.

Color-space rules:

```text
Base color and emissive: sRGB
Normal, roughness, metallic, AO: linear
```

Texture paths must be internal to the package. Runtime CDN dependencies are prohibited.

## 14. Texture budgets

Default limits:

```text
Hero critical texture: 2048 px maximum
Supporting texture:     1024 px maximum
Proxy/thumbnail:          512 px maximum
```

Exceptions require:

- visible quality justification;
- memory estimate;
- lower-resolution variant;
- performance approval.

## 15. Compression

Approved runtime compression may include:

- Meshopt;
- Draco where decode trade-offs are acceptable;
- KTX2/Basis Universal textures;
- standard GLB buffer packing.

The pipeline must record required decoder capabilities in the manifest.

Compression must not introduce visible deformation, broken normals, unstable animation or excessive decode latency.

## 16. Animation contract

Animations must:

- use stable clip names;
- target declared nodes;
- have clean start/end states;
- avoid unnecessary baked keys;
- define looping behavior;
- support interruption where the runtime expects it;
- declare reduced-motion and lower-LOD fallbacks.

Examples:

```text
power_on
power_off
brush_spin
lidar_spin
screen_wake
explode
reassemble
```

Animation clips do not decide when they run. Runtime capability/state mapping owns triggers.

## 17. State-mapping metadata

A model may expose semantic bindings:

```yaml
bindings:
  power:
    material: bulb_material
    property: emissiveIntensity
  color:
    material: bulb_material
    property: emissiveColor
  brightness:
    lightAnchor: bulb_emitter
  cleaning:
    animation: brush_spin
  charging:
    node: status_led
```

Bindings reference canonical capabilities, not raw Home Assistant entity IDs.

## 18. Camera metadata

Required or generated metadata may include:

- hero anchor;
- look-at anchor;
- preferred yaw and pitch;
- inspection limits;
- min/max distance;
- visual-mass offset;
- safe padding;
- exploded-view bounds.

The pipeline validates camera metadata against actual bounds and rejects impossible framing hints.

## 19. Collision and interaction metadata

Optional collision meshes may be provided for:

- picking;
- component selection;
- inspection limits;
- map/room placement;
- camera collision guards.

Collision geometry must be simpler than visual geometry.

Per-part interaction metadata declares:

```ts
interface InteractionNodeMetadata {
  nodeId: string;
  role: "selectable" | "draggable" | "indicator" | "anchor" | "none";
  capabilityIds?: string[];
  accessibleLabel?: string;
}
```

## 20. Model manifest

Each model package includes a versioned manifest.

```yaml
schema_version: 1
id: generic.robot_vacuum.round
version: 1.0.0
display_name: Generic Robot Vacuum
category: vacuum
manufacturer: generic
product_family: round_robot_vacuum
license: CC-BY-4.0
source_attribution: "..."
files:
  lod0: model-lod0.glb
  lod1: model-lod1.glb
  lod2: model-lod2.glb
  proxy: proxy.glb
thumbnail: thumbnail.webp
capabilities:
  - power
  - cleaning
  - battery
camera_preset: vacuum.hero.default
bindings_file: bindings.yaml
integrity:
  algorithm: sha256
  files: {}
```

## 21. Licensing and provenance

Every asset must record:

- author or source;
- license identifier;
- redistribution rights;
- modification rights;
- attribution text where required;
- date acquired;
- original source reference;
- whether trademarks or product likeness restrictions apply.

Assets with unclear redistribution rights must not ship in public model packs.

User-supplied private assets may remain local but still require a local provenance record.

## 22. Exact product likeness

Manufacturer-specific models are optional.

Rules:

- never imply official manufacturer endorsement without authorization;
- use generic fallbacks when rights are unclear;
- separate factual product matching from asset licensing;
- permit users to supply private models locally;
- model packs may be distributed independently from core.

## 23. Validation severity

```text
ERROR   — package cannot register
WARNING — package may register with degraded features
INFO    — optimization or quality suggestion
```

Hard errors include:

- invalid GLB;
- unsupported required extension;
- missing runtime file;
- invalid scale or bounds;
- broken node references;
- missing license metadata for public packs;
- integrity mismatch;
- NaN transforms;
- package ID/version conflict;
- resource cost above hard safety limits without fallback.

## 24. Automated validation

Validation checks:

1. GLB structural validity;
2. schema validity;
3. unit and axis compliance;
4. root hierarchy;
5. bounds and pivot;
6. geometry integrity;
7. material compatibility;
8. texture dimensions and color space;
9. animation targets and clip naming;
10. LOD alignment;
11. semantic binding references;
12. camera metadata;
13. package integrity hashes;
14. licensing fields;
15. estimated transfer, decode and GPU cost.

## 25. Visual validation

Automated checks are followed by rendered review scenes:

```text
Neutral studio
Dark production environment
Bright diagnostic environment
Hero camera
Room-supporting camera
Phone portrait safe frame
Exploded view where supported
```

Review verifies:

- silhouette;
- shading;
- scale;
- pivot;
- emission;
- camera framing;
- transparent materials;
- LOD transitions;
- state mappings.

## 26. Runtime registration

Only validated packages are registered.

```text
Package discovered
→ manifest parsed
→ compatibility checked
→ integrity verified
→ registry entry created
→ assets remain lazy
```

Registration does not decode all models.

## 27. Loading contract

Runtime loading order:

```text
Resolve registry entry
→ select quality tier and LOD
→ show proxy
→ fetch local asset
→ decode off the critical input path
→ validate expected bounds/anchors
→ upload GPU resources
→ match proxy pose
→ reveal final model
```

A failed exact model falls back to category or abstract placeholder without blanking the stage.

## 28. Custom user uploads

User upload flow:

```text
Upload
→ local validation
→ preview
→ normalization options
→ metadata review
→ local package generation
→ registry entry
```

Raw uploads never bypass validation.

The user may choose:

- automatic scale correction;
- pivot correction;
- generic capability mapping;
- private local-only storage.

## 29. Model-pack updates

Updates must preserve:

- stable package ID;
- semantic node IDs where possible;
- compatible capability bindings;
- migration notes for breaking changes;
- previous version rollback data.

Semantic versioning:

```text
PATCH — visual/optimization fix
MINOR — compatible variants, LODs or bindings
MAJOR — breaking node, manifest or capability changes
```

## 30. Security

The pipeline treats model packs as untrusted input.

Rules:

- no executable scripts in asset packages;
- no external URLs loaded automatically;
- strict archive path validation;
- size and decompression limits;
- schema whitelisting;
- integrity hashes;
- shader extensions restricted to approved plugin APIs;
- metadata rendered as text, never executable markup.

## 31. Diagnostics

Each asset exposes:

- registry resolution reason;
- package/version;
- selected LOD;
- compressed and decoded size;
- triangle and draw-call estimates;
- texture memory estimate;
- active animations;
- missing anchors/bindings;
- validation warnings;
- load/decode/upload timings;
- fallback reason.

## 32. CI integration

Public model packs must run:

```text
Manifest schema validation
GLB validation
Budget validation
Integrity validation
License metadata validation
Thumbnail generation/check
Golden render comparison where practical
```

A failed hard check blocks release.

## 33. Acceptance criteria

3.5.7 is approved when:

1. GLB is the only canonical production model format;
2. units, axes, scale, pivot and hierarchy are deterministic;
3. geometry, materials, textures and animation rules are defined;
4. LOD requirements align with runtime performance tiers;
5. canonical capability bindings are independent of HA entity IDs;
6. camera and interaction metadata are versioned;
7. every public asset has explicit provenance and redistribution rights;
8. malformed or unsafe packages cannot register;
9. custom uploads pass through the same validation boundary;
10. exact-model failure always resolves to a usable fallback;
11. CI can automatically reject invalid model packs;
12. diagnostics explain resolution, cost and fallback behavior.
