# 3.5.8 — Plugin and Extension API

**Status:** Approved engineering baseline  
**Phase:** 3.5 — Engineering Specifications  
**Depends on:** 3.5.1 Runtime Execution Model, 3.5.5 Home Assistant Runtime Contract, 3.5.6 Performance and Resource Budgets, 3.5.7 3D Asset Pipeline and Validation

## 1. Purpose

The Plugin and Extension API defines how third parties extend Home Configurator without modifying core runtime code.

Extensions may add device profiles, capability resolvers, model packs, themes, blueprints, animations, shaders, renderer features, editor panels, diagnostics and Home Assistant adapters. They may not bypass runtime ownership, mutate core state directly, call Home Assistant services outside the command system, or write arbitrary objects into the live scene graph.

## 2. Extension principles

1. Core remains usable without any third-party package.
2. Extensions register declarative capabilities through versioned interfaces.
3. Runtime ownership remains with core services.
4. User overrides always win over plugin defaults.
5. One failing plugin must not crash the application.
6. Packages operate locally by default and must not require cloud services.
7. Permissions are explicit and least-privilege.
8. Compatibility is checked before load.
9. Every extension is unloadable.
10. Unknown data is preserved where safe but never executed.

## 3. Supported package types

```text
plugin
model-pack
theme-pack
blueprint-pack
animation-pack
shader-pack
environment-pack
integration-adapter
developer-tool
```

A repository may contain more than one package, but each published package has one manifest and one primary type.

## 4. Package manifest

```ts
interface ExtensionManifest {
  schemaVersion: string;
  id: string;
  name: string;
  version: string;
  type: ExtensionPackageType;
  description?: string;
  author: string;
  license: string;
  homepage?: string;
  repository?: string;
  entry?: string;
  assets?: string[];
  minimumCore: string;
  maximumCore?: string;
  dependencies?: ExtensionDependency[];
  optionalDependencies?: ExtensionDependency[];
  permissions?: ExtensionPermission[];
  contributes?: ExtensionContributionManifest;
  integrity?: string;
}
```

Package IDs use reverse-domain or stable scoped naming and must remain immutable after publication.

## 5. Compatibility

Before loading, the package manager validates:

- manifest schema version;
- core version range;
- dependency versions;
- required runtime capabilities;
- package integrity;
- duplicate contribution IDs;
- prohibited permissions;
- asset validation status.

Incompatible packages remain installed but disabled with a clear reason.

## 6. Lifecycle

```text
DISCOVERED
→ VALIDATED
→ RESOLVED
→ LOADING
→ INITIALIZING
→ REGISTERING
→ READY
→ SUSPENDED
→ UNLOADING
→ DISPOSED
```

Failure states:

```text
INVALID
INCOMPATIBLE
DEPENDENCY_FAILED
LOAD_FAILED
RUNTIME_FAILED
QUARANTINED
```

Lifecycle interface:

```ts
interface HomeConfiguratorExtension {
  manifest: ExtensionManifest;
  initialize(context: ExtensionContext): Promise<void> | void;
  register(registry: ExtensionRegistry): Promise<void> | void;
  start?(): Promise<void> | void;
  suspend?(): Promise<void> | void;
  resume?(): Promise<void> | void;
  dispose(): Promise<void> | void;
}
```

`dispose()` is mandatory and must release listeners, timers, GPU resources, subscriptions and registry handles.

## 7. Extension context

Plugins receive a capability-limited context rather than internal singleton access.

```ts
interface ExtensionContext {
  readonly coreVersion: string;
  readonly packageId: string;
  readonly logger: ExtensionLogger;
  readonly events: ExtensionEventApi;
  readonly commands: ExtensionCommandApi;
  readonly state: ReadonlyExtensionStateApi;
  readonly assets: ExtensionAssetApi;
  readonly diagnostics: ExtensionDiagnosticsApi;
  readonly capabilities: ReadonlySet<ExtensionPermission>;
}
```

The context never exposes mutable core stores, the Three.js renderer instance, raw Home Assistant authentication data, or unrestricted browser storage.

## 8. Contribution registries

Extensions register contributions through typed registries.


Every registration returns a disposable handle.

```ts
interface RegistrationHandle {
  readonly id: string;
  dispose(): void;
}
```

## 9. Device-profile extensions

A device plugin may contribute:

- manufacturer and model matchers;
- entity-composition rules;
- capability detectors;
- semantic commands;
- preferred controls;
- model recommendations;
- state-to-animation mappings;
- optional editor metadata.

It must not hardcode a user entity ID.

```ts
interface DeviceProfileContribution {
  id: string;
  match(input: DeviceMatchInput): MatchResult;
  compose(input: DeviceCompositionInput): ResolvedDevicePatch;
  capabilities?: CapabilityContribution[];
  commands?: SemanticCommandContribution[];
  presentation?: DevicePresentationHints;
}
```

Matching is confidence-scored and user overrides remain authoritative.

## 10. Capability resolvers

Resolvers may infer canonical capabilities from normalized Home Assistant data.

Rules:

- evidence must be reported;
- confidence must be bounded from `0..1`;
- a resolver cannot silently replace a higher-confidence explicit capability;
- conflicting results are retained for diagnostics and editor review;
- raw service calls are not emitted from detection code.

## 11. Command extensions

Extensions contribute semantic command translators, never direct UI service calls.

```ts
interface SemanticCommandContribution {
  id: string;
  capability: string;
  supports(device: ResolvedDevice): boolean;
  build(intent: SemanticIntent, device: ResolvedDevice): CommandPlan;
}
```

All command plans pass through core validation, queueing, optimistic state, retry and reconciliation.

## 12. Renderer and scene extensions

Renderer extensions are restricted to registered hooks and owned scene nodes.

Allowed:

- custom materials derived from approved material interfaces;
- validated shader modules;
- extension-owned scene subtrees;
- post-processing passes that declare cost and fallback;
- registered animations and transitions.

Prohibited:

- replacing the core renderer;
- mutating the active camera;
- writing directly to nodes owned by another package;
- starting independent frame loops;
- bypassing quality tiers;
- allocating untracked GPU resources.

```ts
interface RenderContribution {
  id: string;
  qualityMinimum: "full" | "balanced" | "essential";
  estimatedCost: ResourceCostDeclaration;
  create(context: RenderExtensionContext): DisposableRenderFeature;
}
```

## 13. Shader extensions

Shaders must declare:

- stage and purpose;
- uniforms and defaults;
- supported quality tiers;
- reduced-motion behavior;
- texture and render-target requirements;
- maximum passes;
- deterministic fallback;
- license and source attribution.

Shader compilation failure disables only that feature and activates its fallback.

## 14. Model packs

Model packs contain validated assets and registry metadata only. They do not execute JavaScript unless separately published as a plugin.

A model pack may provide:

- exact models;
- generic models;
- variants;
- LODs;
- thumbnails;
- camera anchors;
- animation clips;
- material bindings;
- capability-to-node mappings.

Every model must pass the 3.5.7 asset validator.

## 15. Themes and blueprints

Theme packages may modify design tokens and approved presentation resources. They cannot change application logic.

Blueprint packages may declare layout, navigation, presentation, motion and placeholder intent, but cannot embed executable code.

Unknown blueprint extension fields are namespaced by package ID.

## 16. Editor contributions

Editor plugins may add:

- property inspectors;
- mapping helpers;
- validation panels;
- model-preview tools;
- diagnostics views;
- import/export helpers.

Runtime pages must not depend on editor-only plugins.

## 17. Permissions

Baseline permission vocabulary:

```text
state.read
commands.propose
events.subscribe
assets.read
assets.register
models.register
themes.register
blueprints.register
renderer.feature
shader.register
editor.extend
diagnostics.publish
storage.local
network.external
```

`network.external` is denied by default and requires explicit user approval. Home Assistant authentication secrets are never exposed to plugins.

## 18. Storage

Plugins may use namespaced local storage through the extension context.

Rules:

- quota is enforced;
- data is scoped to package ID;
- uninstall offers data removal;
- migrations are versioned;
- plugins cannot inspect another package's storage;
- sensitive data must not be stored without explicit purpose and encryption support.

## 19. Event access

Plugins subscribe only to public, versioned events.

Examples:

```text
runtime.ready
room.changed
device.selected
capability.updated
command.pending
command.confirmed
command.failed
quality.changed
extension.suspended
```

Plugins cannot intercept or cancel unrelated core events unless the event contract explicitly supports a proposal phase.

## 20. Failure isolation

Each plugin is wrapped by an error boundary.

On failure:

1. the failing contribution is disabled;
2. owned registrations are disposed;
3. core fallbacks remain active;
4. the error is recorded in diagnostics;
5. repeated failures may quarantine the package;
6. unrelated plugins continue running.

A renderer extension that repeatedly exceeds budget may be suspended automatically.

## 21. Dependency handling

Dependencies are resolved as a directed acyclic graph.

Rules:

- cycles invalidate the affected packages;
- required dependency failure blocks load;
- optional dependency failure degrades features only;
- load order follows dependency order, then stable package ID;
- uninstall is blocked or confirmed when dependants exist.

## 22. Hot reload and updates

Development mode may support hot reload. Production updates use controlled unload/reload:

```text
validate new package
→ suspend old package
→ migrate package storage
→ unload old registrations
→ load new package
→ verify health
→ commit update or rollback
```

No update may leave duplicate registrations active.

## 23. Security boundary

The first production release treats third-party executable plugins as trusted local code installed deliberately by the user. The API still enforces capability boundaries, but it is not a perfect security sandbox.

Therefore:

- executable packages are clearly distinguished from data-only packs;
- package source and permissions are shown before installation;
- integrity hashes are supported;
- remote code loading at runtime is prohibited;
- `eval`, dynamic script injection and unverified CDN dependencies are prohibited;
- security-sensitive contributions may be disabled in safe mode.

## 24. Safe mode

Safe mode loads:

- core runtime;
- official bundled assets;
- user configuration;
- no third-party executable plugins.

It is activated after repeated startup failure or by explicit user action.

## 25. Versioning

The extension API follows semantic versioning.

- patch: compatible fixes;
- minor: backward-compatible additions;
- major: breaking contract change.

Deprecated APIs remain available for at least one documented compatibility window where practical.

## 26. Developer SDK

The SDK must provide:

- TypeScript types;
- manifest schema;
- test harness;
- mock Home Assistant runtime;
- mock state and command APIs;
- asset validator integration;
- renderer-budget test utilities;
- example packages;
- compatibility checker;
- package build command.

## 27. Diagnostics

Developer mode exposes:

- package version and state;
- lifecycle phase;
- registered contributions;
- declared permissions;
- dependency graph;
- event subscriptions;
- command proposals;
- GPU and CPU cost;
- storage use;
- errors and quarantine reason;
- unload/disposal leaks.

## 28. Public package-manager interface

```ts
interface ExtensionManager {
  discover(): Promise<ExtensionManifest[]>;
  validate(manifest: ExtensionManifest): Promise<ValidationResult>;
  install(source: PackageSource): Promise<void>;
  enable(packageId: string): Promise<void>;
  disable(packageId: string): Promise<void>;
  update(packageId: string): Promise<void>;
  uninstall(packageId: string, removeData?: boolean): Promise<void>;
  list(): ReadonlyArray<ExtensionStatus>;
}
```

## 29. Acceptance criteria

3.5.8 is approved when:

1. Every extension type has a defined registration path.
2. Plugins cannot mutate core state, camera or renderer directly.
3. Commands always pass through the semantic command queue.
4. Lifecycle and disposal are deterministic.
5. Compatibility and dependencies are validated before load.
6. Permissions are explicit and least-privilege.
7. One failing plugin cannot crash the runtime.
8. Data-only packs remain non-executable.
9. Renderer and shader contributions declare cost and fallback.
10. User overrides remain authoritative.
11. Safe mode can start without third-party executable code.
12. Package updates can rollback without duplicate registrations.
13. Diagnostics can identify contribution ownership and resource leaks.
14. A contributor can build a device plugin without importing private core modules.
