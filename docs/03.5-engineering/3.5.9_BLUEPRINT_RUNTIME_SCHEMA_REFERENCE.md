# 3.5.9 — Blueprint and Runtime Schema Reference

**Status:** Approved engineering baseline  
**Phase:** 3.5 — Engineering Specifications  
**Depends on:** 2.4 Data Model, 2.8 Blueprint Format, 3.5.1 Runtime Execution Model, 3.5.5 Home Assistant Runtime Contract, 3.5.8 Plugin and Extension API

## 1. Purpose

This specification defines the canonical, versioned schema used to describe portable Home Configurator blueprints and the resolved runtime configuration produced after discovery, mapping and validation.

A blueprint must remain shareable across installations. It describes intent, capabilities, layout and presentation without hard-coding a user's Home Assistant entity IDs.

## 2. Ownership boundary

The Schema Engine owns:

- schema validation;
- normalization;
- migrations;
- placeholder resolution;
- runtime materialization;
- import and export safety;
- compatibility checks;
- schema diagnostics.

It does not own:

- Home Assistant discovery;
- capability detection;
- renderer behavior;
- plugin execution;
- user-interface layout calculations;
- command dispatch.

## 3. Document classes

```text
BlueprintDocument
RuntimeConfiguration
MappingRecord
OverrideDocument
PackageReference
MigrationRecord
ValidationReport
```

### BlueprintDocument
Portable, shareable and entity-independent.

### RuntimeConfiguration
Installation-specific and fully resolved. It may contain entity IDs and internal registry identifiers, but is never exported as a portable blueprint by default.

### MappingRecord
Stores how a placeholder was matched to a discovered room, device, entity set or capability provider.

### OverrideDocument
Stores user changes separately from imported blueprint content so updates do not overwrite local decisions.

## 4. Top-level blueprint schema

```ts
interface BlueprintDocument {
  schemaVersion: string;
  id: string;
  version: string;
  metadata: BlueprintMetadata;
  compatibility: CompatibilityRequirement;
  requirements?: BlueprintRequirements;
  theme?: ThemeReference;
  environment?: EnvironmentReference;
  navigation?: NavigationBlueprint;
  rooms?: RoomBlueprint[];
  devices?: DevicePlaceholder[];
  systems?: SystemBlueprint[];
  scenes?: SceneBlueprint[];
  models?: ModelReference[];
  camera?: CameraBlueprint;
  motion?: MotionBlueprint;
  quickActions?: QuickActionBlueprint[];
  packages?: PackageReference[];
  extensions?: Record<string, unknown>;
}
```

Unknown top-level properties are rejected unless they are namespaced under `extensions`.

## 5. Metadata

```ts
interface BlueprintMetadata {
  name: string;
  description?: string;
  author?: string;
  license?: string;
  homepage?: string;
  tags?: string[];
  createdAt?: string;
  updatedAt?: string;
}
```

Metadata must not contain Home Assistant secrets, URLs with access tokens or private entity snapshots.

## 6. Schema versioning

`schemaVersion` uses semantic versioning.

```text
MAJOR — breaking schema change
MINOR — backward-compatible fields or capabilities
PATCH — clarifications or validation corrections
```

Runtime behavior:

- newer compatible minor versions may load after validation;
- unsupported major versions are rejected before materialization;
- migration is explicit and recorded;
- original input is preserved until migration succeeds;
- no destructive in-place migration occurs without a recoverable backup.

## 7. Compatibility contract

```ts
interface CompatibilityRequirement {
  minimumCore: string;
  maximumCore?: string;
  requiredPackages?: PackageRequirement[];
  optionalPackages?: PackageRequirement[];
  requiredCapabilities?: string[];
}
```

Missing optional packages degrade gracefully. Missing required packages prevent activation but not inspection.

## 8. Entity-independent placeholders

Blueprints express semantic roles rather than entity IDs.

```ts
interface DevicePlaceholder {
  id: string;
  role?: string;
  profile?: string;
  roomRef?: string;
  requiredCapabilities: string[];
  optionalCapabilities?: string[];
  manufacturerHint?: string;
  modelHint?: string;
  multiplicity?: "one" | "zero-or-one" | "one-or-more" | "many";
  preferredModelRef?: string;
  presentation?: DevicePresentationBlueprint;
}
```

Example:

```yaml
id: living-room-primary-light
role: primary-light
profile: rgb-light
roomRef: living-room
requiredCapabilities:
  - power
  - brightness
optionalCapabilities:
  - color.rgb
  - color.temperature
```

Portable blueprints must not require `light.living_room_lamp` or any other installation-specific identifier.

## 9. Room placeholders

```ts
interface RoomBlueprint {
  id: string;
  name?: string;
  role?: string;
  floorRef?: string;
  match?: RoomMatchRule;
  deviceRefs?: string[];
  defaultDeviceRef?: string;
  cameraPresetRef?: string;
  order?: number;
  presentation?: RoomPresentationBlueprint;
}
```

Room matching may use:

- explicit user assignment;
- area name hints;
- area aliases;
- contained device profiles;
- floor metadata;
- discovery confidence.

Name similarity alone must never silently finalize a low-confidence mapping.

## 10. Matching and mapping

```ts
interface MappingRecord {
  placeholderId: string;
  targetType: "room" | "device" | "entity" | "system" | "scene";
  targetId: string;
  confidence: number;
  source: "user" | "exact" | "profile" | "capability" | "heuristic";
  evidence: string[];
  confirmed: boolean;
  updatedAt: string;
}
```

Confidence bands:

```text
0.90–1.00  high
0.65–0.89  medium
0.00–0.64  low
```

Rules:

- high-confidence exact matches may auto-apply when policy permits;
- medium-confidence matches require review by default;
- low-confidence matches remain unresolved;
- user-confirmed mappings remain authoritative until explicitly changed;
- remapping does not modify the source blueprint.

## 11. Resolution pipeline

```text
Parse
→ Structural validation
→ Schema migration
→ Package compatibility validation
→ Normalize defaults
→ Discover installation candidates
→ Resolve room placeholders
→ Resolve device placeholders
→ Resolve capability providers
→ Apply user mappings
→ Apply local overrides
→ Validate runtime references
→ Materialize RuntimeConfiguration
→ Publish validation report
```

No renderer or controller receives an unvalidated blueprint object.

## 12. Runtime configuration

```ts
interface RuntimeConfiguration {
  schemaVersion: string;
  sourceBlueprintId?: string;
  sourceBlueprintVersion?: string;
  installationId: string;
  generatedAt: string;
  rooms: ResolvedRoomConfiguration[];
  devices: ResolvedDeviceConfiguration[];
  systems: ResolvedSystemConfiguration[];
  scenes: ResolvedSceneConfiguration[];
  navigation: ResolvedNavigationConfiguration;
  theme: ResolvedThemeConfiguration;
  camera: ResolvedCameraConfiguration;
  packages: ResolvedPackageReference[];
  mappings: MappingRecord[];
  diagnostics: RuntimeConfigurationDiagnostics;
}
```

Runtime configuration may reference Home Assistant entity and device IDs because it is local, installation-specific state.

## 13. Override precedence

```text
Core defaults
< Blueprint values
< Package defaults
< Installation mappings
< User overrides
< Session-only preview state
```

Session preview state is never persisted automatically.

User overrides are stored separately and survive compatible blueprint upgrades.

## 14. Override schema

```ts
interface OverrideDocument {
  schemaVersion: string;
  blueprintId?: string;
  installationId: string;
  roomOverrides?: Record<string, Partial<ResolvedRoomConfiguration>>;
  deviceOverrides?: Record<string, Partial<ResolvedDeviceConfiguration>>;
  navigationOverrides?: Partial<ResolvedNavigationConfiguration>;
  themeOverrides?: Record<string, unknown>;
  mappingOverrides?: MappingRecord[];
}
```

Overrides may alter supported presentation and mapping properties but may not inject executable code.

## 15. Package and extension references

```ts
interface PackageReference {
  id: string;
  versionRange: string;
  required?: boolean;
  configuration?: Record<string, unknown>;
}
```

Extension configuration must be namespaced:

```yaml
extensions:
  com.example.package:
    option: value
```

The owning plugin validates its extension block through the registered schema API from 3.5.8.

## 16. Import behavior

Import sequence:

1. read without executing extensions;
2. validate file type and size;
3. parse in a restricted data-only path;
4. validate schema version;
5. list package dependencies and permissions;
6. show migration requirements;
7. preview mappings and unresolved placeholders;
8. require user confirmation before activation;
9. preserve the previous active configuration for rollback.

Import must never dispatch Home Assistant services.

## 17. Export levels

```text
THEME_ONLY
PRESENTATION
LAYOUT_AND_NAVIGATION
INTERACTION
FULL_PORTABLE_BLUEPRINT
LOCAL_BACKUP
```

`FULL_PORTABLE_BLUEPRINT` strips:

- entity IDs;
- device registry IDs;
- installation identifiers;
- access tokens;
- private URLs;
- cached states;
- vacuum map images and coordinates;
- local usage history.

`LOCAL_BACKUP` may include mappings and IDs but must be clearly marked non-portable and sensitive.

## 18. Validation report

```ts
interface ValidationReport {
  valid: boolean;
  errors: ValidationIssue[];
  warnings: ValidationIssue[];
  migrations: MigrationRecord[];
  unresolvedPlaceholders: string[];
  missingRequiredPackages: string[];
  missingOptionalPackages: string[];
}
```

Each issue contains:

- stable code;
- severity;
- JSON path;
- user-facing message;
- technical detail;
- suggested recovery where available.

## 19. Failure behavior

### Invalid syntax

- reject activation;
- preserve current runtime;
- identify line/path when available.

### Unsupported schema major version

- allow inspection;
- block activation;
- offer supported migration only when registered.

### Missing required package

- preserve blueprint in inactive state;
- list exact dependency;
- do not silently remove dependent sections.

### Missing optional package

- disable only affected contribution;
- use registered fallback where possible.

### Unresolved required placeholder

- keep the blueprint in review state;
- do not invent a mapping.

### Invalid local override

- isolate the invalid override;
- retain the valid blueprint and other overrides;
- expose recovery in diagnostics/editor.

## 20. Migrations

```ts
interface SchemaMigration {
  from: string;
  to: string;
  migrate(document: unknown): unknown;
  validateResult(document: unknown): ValidationReport;
}
```

Migration requirements:

- deterministic;
- idempotent where practical;
- pure data transformation;
- no network calls;
- no Home Assistant service calls;
- tested with fixtures;
- migration chain recorded;
- rollback source retained.

## 21. Serialization rules

Canonical JSON is the normative storage format.

YAML may be accepted as an authoring/import format but must normalize to the same canonical data model.

Rules:

- UTF-8 only;
- stable IDs required;
- dates use ISO 8601;
- no functions or executable expressions;
- no cyclic references;
- references use IDs, never object pointers;
- numeric units are explicit where ambiguity exists;
- unknown fields outside namespaced extension blocks fail validation.

## 22. Security and privacy

- schemas are data-only;
- imported documents cannot execute code;
- URL fields are allowlisted by purpose;
- secrets are prohibited;
- exports are scrubbed according to export level;
- sensitive local backups require explicit confirmation;
- diagnostics redact entity names and identifiers when shared externally unless the user opts in.

## 23. Diagnostics

Developer mode exposes:

- active schema version;
- source blueprint and version;
- migration chain;
- placeholder resolution status;
- mapping confidence and evidence;
- override precedence trace;
- unresolved references;
- package schema registrations;
- validation timing;
- last successful runtime materialization.

## 24. Public engine contract

```ts
interface BlueprintSchemaEngine {
  parse(input: string, format: "json" | "yaml"): unknown;
  validate(document: unknown): ValidationReport;
  migrate(document: unknown, targetVersion?: string): MigrationResult;
  resolve(document: BlueprintDocument, context: ResolutionContext): Promise<ResolutionResult>;
  materialize(result: ResolutionResult): RuntimeConfiguration;
  exportBlueprint(runtime: RuntimeConfiguration, options: ExportOptions): BlueprintDocument;
}
```

## 25. Required tests

Implementation must include tests for:

- valid minimal blueprint;
- valid full blueprint;
- unknown root fields;
- namespaced extension validation;
- major-version rejection;
- multi-step migration;
- exact, medium and low-confidence mappings;
- unresolved required placeholders;
- optional package fallback;
- override precedence;
- blueprint upgrade preserving user overrides;
- portable export scrubbing local identifiers;
- local backup classification;
- rollback after failed activation;
- malformed and oversized imports.

## 26. Acceptance criteria

3.5.9 is approved when:

1. Portable blueprints contain no required installation-specific entity IDs.
2. Runtime configuration is distinct from blueprint source data.
3. Schema and package compatibility are validated before activation.
4. Placeholder resolution is deterministic and confidence-aware.
5. User-confirmed mappings and overrides remain authoritative.
6. Unknown fields are rejected except inside registered namespaced extensions.
7. Migrations are pure, versioned, recorded and recoverable.
8. Imports cannot execute code or dispatch Home Assistant services.
9. Portable exports scrub private installation data.
10. Missing optional content degrades locally without invalidating the whole blueprint.
11. Validation reports identify exact paths and recovery actions.
12. A complete runtime configuration can be materialized without feature modules inventing schema behavior.
