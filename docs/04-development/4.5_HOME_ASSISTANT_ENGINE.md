# 4.5 — Home Assistant Engine

**Status:** Complete and CI validated  
**Milestone version:** 0.5.0

## Delivered

- authenticated Home Assistant WebSocket transport;
- request, response and event-subscription correlation;
- connection, authentication, sync, degraded, reconnecting and disconnected states;
- bounded exponential reconnect with jitter;
- coherent initial area, floor, device, entity, state, service and configuration synchronization;
- automatic room and device discovery without per-entity dashboard configuration;
- semantic capability detection for lights, switches, climate, media players, vacuums, covers, locks and sensors;
- stable canonical room, device and entity identifiers;
- composed-device availability that preserves working capabilities when supporting entities fail;
- incremental `state_changed` processing;
- registry-change subscriptions and refresh;
- semantic command translation into Home Assistant service calls;
- optimistic state overlays;
- acknowledgement versus authoritative confirmation separation;
- continuous-command coalescing;
- command timeout, failure and reconciliation handling;
- stale-state behavior and command rejection while disconnected;
- runtime adapter compatible with the existing Runtime Core boundary;
- in-memory Home Assistant transport for tests and local demonstrations;
- dashboard demonstration containing automatically discovered rooms, lights, TV, Apple TV, Roborock vacuum, thermostat and switches;
- diagnostics for status, registries, subscriptions, events, reconnect attempts and commands.

## Runtime flow

```text
Home Assistant WebSocket
→ authentication
→ registry and state synchronization
→ canonical room/device discovery
→ capability bindings
→ confirmed runtime snapshot
→ dashboard, renderer and controls
```

Commands flow back through one boundary:

```text
semantic command
→ capability translation
→ optimistic state
→ Home Assistant service call
→ acknowledgement
→ authoritative state event
→ confirmation or reconciliation
```

## Security boundary

The access token is used only by the WebSocket transport and is never included in diagnostics. The dashboard demonstration uses the in-memory transport and contains no real credentials. Native Home Assistant deployment may supply an authenticated transport instead of requesting a long-lived token.
