# 4.7.2.2 — Home Assistant State Adapter

**Status:** Implementation complete; CI validation in progress  
**Milestone version:** 0.7.3  
**Pull request:** #21

## Scope

Connect the canonical Home Assistant runtime to the application-owned `DeviceStore` introduced in 4.7.2.1. Home Assistant remains an external state producer; Runtime remains the integration-neutral source of confirmed, optimistic and effective device state for future UI and graphics consumers.

## Delivered

- `HomeAssistantStateAdapter` implementing the existing Home Assistant runtime contract;
- confirmed snapshot translation into Runtime device descriptors and semantic state;
- normalised power, brightness, colour, colour temperature, climate, media, cover, lock and sensor values;
- optimistic Runtime patches derived from semantic commands;
- authoritative confirmation when Home Assistant state updates arrive;
- rollback for rejected, offline and locally timed-out commands;
- command lifecycle events for dispatching, acknowledgement, confirmation, failure, cancellation and timeout;
- Home Assistant connectivity mapped separately from device availability;
- structural synchronisation for registration, metadata changes and removed devices;
- dashboard integration through the Runtime `DeviceStore`;
- Runtime-state diagnostics for device count, connectivity and optimistic state;
- integration tests covering confirmed state, optimistic reconciliation, colour and brightness conversion, disconnection and rollback.

## State boundary

The adapter depends on both `@home-configurator/home-assistant` and `@home-configurator/runtime`, but it does not introduce a dependency from Runtime back to Home Assistant. UI and Graphics remain independent from Home Assistant-specific state and will bind to `DeviceStore.effectiveState` in the next slice.
