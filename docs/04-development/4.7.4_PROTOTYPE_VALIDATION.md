# 4.7.4 — Prototype Validation and Device Testing

**Status:** In progress  
**Branch:** `feature/prototype-validation-v0.7.5`

## Scope

This milestone closes the first interactive prototype through automated and manual validation rather than adding new product capability.

## Automated gates

- viewport classification for phone, tablet portrait, tablet landscape and desktop;
- reduced-motion compatibility;
- keyboard navigation availability;
- accessible naming for the interactive stage and selected device;
- generic fallback-model availability;
- runtime diagnostics availability;
- full formatting, lint, strict TypeScript, tests and production builds.

## Manual acceptance matrix

| Surface | Required acceptance |
|---|---|
| iPad Safari portrait | Room and device navigation remain reachable; configurator does not obscure the selected model; two-finger vertical brightness works. |
| iPad Safari landscape | Model, navigation and configurator fit without horizontal overflow; colour drag and brightness gestures remain stable. |
| Desktop Safari/Chrome | Mouse colour drag, wheel brightness and keyboard controls work; focus remains visible. |
| Phone portrait | Navigation and configurator use the mobile presentation without clipped controls or unsafe-area overlap. |
| VoiceOver | Stage, navigation, selected device and controls expose useful names; state changes are announced without duplicate chatter. |
| Reduced motion | Camera and panel transitions complete without looping float or unnecessary spatial motion. |
| Offline/unavailable device | Controls reject unsafe commands, the model communicates unavailability and the interface remains navigable. |
| Generic device fallback | Unknown devices render the fallback asset without breaking selection or diagnostics. |

## Real Home Assistant acceptance

1. Connect one colour-capable light.
2. Confirm discovery, room assignment and availability.
3. Change power, colour and brightness from the prototype.
4. Confirm optimistic visual feedback appears immediately.
5. Confirm authoritative Home Assistant state replaces optimistic state.
6. Force one rejected or timed-out command and confirm rollback.
7. Change the light externally in Home Assistant and confirm the selected model updates.
8. Disconnect and reconnect Home Assistant and confirm stale state is communicated safely.

## Exit criteria

The milestone is complete only when automated CI passes and the manual matrix has no critical accessibility, interaction, responsive or state-reconciliation defects.
