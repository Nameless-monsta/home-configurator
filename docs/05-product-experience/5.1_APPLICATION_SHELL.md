# 5.1 — Application Shell

**Status:** Active

## Objective

Build the real, production-bound application shell around the completed Runtime, Home Assistant, graphics and interaction foundations.

This is not a disposable prototype. It is the first visually faithful vertical slice of the product.

## Experience target

The shell should feel spatial, calm and premium:

- a full-screen scene is the primary surface;
- the selected room and device are visually central;
- navigation floats around the scene instead of occupying permanent dashboard columns;
- device controls appear only when relevant;
- diagnostics and engineering information remain available but secondary;
- typography, glass, depth and motion follow the approved iyO-inspired direction;
- the application remains usable with keyboard input and reduced motion.

## Required vertical slice

```text
Launch application
→ enter Living Room
→ see the spatial scene
→ select IKEA TRÅDFRI lamp
→ reveal contextual controls
→ manipulate colour and brightness
→ dispatch through Runtime and Home Assistant
→ render optimistic feedback
→ render confirmation or rollback
```

## Scope

### Launch and access

- [ ] Confirm the real dashboard entry point.
- [ ] Provide one documented development command.
- [ ] Document the expected local URL.
- [ ] Ensure the build opens on desktop and iPad browsers.
- [ ] Add a preview deployment path or shareable review build.

### Spatial shell

- [ ] Replace the conventional dashboard composition with a full-screen spatial canvas.
- [ ] Add minimal floating home and room context.
- [ ] Add contextual room and device navigation.
- [ ] Keep the selected device as the hero object.
- [ ] Hide technical panels by default.

### Contextual control layer

- [ ] Reveal controls when the lamp is selected.
- [ ] Dismiss controls when the device is deselected.
- [ ] Show power, brightness, colour, availability and pending state.
- [ ] Preserve keyboard and reduced-motion alternatives.

### Runtime integration

- [ ] Use the existing Device Store as the effective-state source.
- [ ] Reuse the existing Home Assistant command and reconciliation path.
- [ ] Reuse selected-device model binding.
- [ ] Preserve optimistic confirmation and rollback behaviour.
- [ ] Keep diagnostics available through a secondary developer surface.

### Review evidence

- [ ] Attach desktop screenshot.
- [ ] Attach iPad landscape screenshot.
- [ ] Attach a short interaction recording or preview URL.
- [ ] Record product-owner feedback and required changes.

## Out of scope

These are later Product Experience milestones:

- final production-quality lamp model and materials;
- the complete floating spatial colour sphere;
- cinematic navigation across multiple finished rooms;
- AI assistant experience;
- broad device-type coverage.

The shell should leave clear integration points for those milestones without attempting to complete them prematurely.

## Acceptance criteria

5.1 is complete only when all of the following are true:

1. The product owner can run or open the real application without reconstructing the setup.
2. The initial screen is recognisably aligned with the approved spatial direction.
3. The Living Room and IKEA lamp journey works end to end.
4. The interface is not dominated by permanent engineering or dashboard panels.
5. Runtime and Home Assistant behaviour remain functional.
6. Desktop and iPad layouts are visibly reviewable.
7. A screenshot set or preview URL is attached to the milestone record.
8. The product owner has reviewed the running experience.

## Validation

Engineering validation:

```bash
pnpm check
```

Product validation:

- open the application on desktop;
- open it on iPad landscape;
- select and control the lamp;
- background and restore the tab;
- test reduced motion;
- test keyboard selection and dismissal;
- compare the running composition with the approved iyO-inspired specification.
